How AWS Lambda Works Case Study

AWS Lambda is Function-as-a-Service (FaaS): you upload a function, and AWS runs it in response to events — an HTTP request, a file upload, a queue message — without you managing any servers. The fascinating system-design problem underneath is how Lambda gives you the illusion of "infinite, instant, pay-per-use compute" by juggling a pool of short-lived sandboxed containers, balancing the cold start cost against keeping containers warm.

Figure: an event triggers the invoke service, which routes to a warm micro-VM sandbox when one is free, or (dashed) creates a brand-new sandbox on a cold start before running your function code.

Requirements (functional & non-functional)

  • Functional: run user code in response to events; isolate each tenant's code securely; scale from zero to thousands of concurrent executions automatically; bill only for actual execution time.
  • Non-functional: strong multi-tenant isolation (your code must never see mine); low startup latency; high density (pack many functions per physical host to be cost-efficient); elastic — scale up in seconds, scale to zero when idle.
Senior tip: the whole design is a tension between two goals — isolation (security wants a fresh, sandboxed environment per tenant) and speed/cost (performance wants to reuse a hot environment). Lambda's architecture is essentially the resolution of that tension.

Step-by-step request flow

Tracing the components in the diagram, here is the life of a single invocation:

  1. Event source: an API Gateway request, S3 object-created event, or queue message arrives and triggers the function.
  2. Invoke service: the front door decides where to run this invocation. It looks for a free, already-initialized environment for this function/version.
  3. Worker fleet (★) — warm path: if a free environment exists, the request is routed to it and execution begins almost immediately (~1–10ms of platform overhead).
  4. Worker fleet (★) — cold path (dashed): if none is free, a new Firecracker micro-VM sandbox is created (boots in ~125ms), the runtime is bootstrapped, and your init code runs — the cold-start cost.
  5. Function code: in either case the sandbox finally invokes your handler. Afterwards the environment is kept warm for a while, then frozen and reclaimed if idle.

The execution environment lifecycle

Lambda runs your code inside an execution environment — a sandboxed micro-VM (AWS uses Firecracker). Its life has distinct phases, and understanding them explains every latency surprise:

  • Init (cold start): download your code, start the sandbox, bootstrap the runtime (Node/Python/etc.), and run your initialization code (everything outside the handler). This is the slow part.
  • Invoke (warm): the handler runs. If the environment already exists from a previous request, you skip Init entirely — this is a warm start and it's fast.
  • Shutdown: after a period of inactivity, AWS freezes then destroys the environment to reclaim resources. The next request after that pays a cold start again.

The core hard problem: cold starts vs warm containers

A request arrives. If there's a free, already-initialized environment, route to it → warm, milliseconds. If none is free, spin up a new one → cold start, tens to hundreds of ms (sometimes seconds with big dependencies or VPC attachment). This is exactly what the demo simulates.

Warm startReuses an existing environment. Fast (~1-10ms overhead). But the environment may hold leftover state from the last invocation — global variables persist!
Cold startFresh, clean environment. Pays full Init cost. Worse for latency-sensitive, spiky traffic.
Provisioned concurrencyPre-warm N environments so they're always ready → no cold starts. Costs money even when idle.
Gotcha: because warm environments are reused, global/module-scope state survives between invocations. Caching a DB connection there is good; accidentally leaking one user's data into a global that the next user reads is a real bug. Treat the handler as stateless; treat globals as a per-environment cache only.

Concurrency & auto-scaling

Concurrency = number of requests executing at the same instant. One environment handles exactly one request at a time. So to serve 100 simultaneous requests, Lambda needs 100 environments. It scales by spinning up environments on demand, up to an account/function limit.

Concrete numbers make the model click. The default account limit is 1,000 concurrent executions per region (raisable to tens of thousands). A useful identity is concurrency ≈ request rate × average duration: 1,000 requests/sec at 50ms each needs only ~50 environments, but 1,000 req/sec at 2s each needs ~2,000 — over the default cap. Lambda scales by adding up to 1,000 new environments every 10 seconds per function (a burst then a ramp), which is why a sudden 0→10,000 spike produces a wave of cold starts rather than instant capacity: provisioning thousands of micro-VMs simultaneously would blow the per-host density and warm-pool budget. A cold start adds anywhere from ~100ms (small Node/Python) to over 1s (large dependencies, or +several hundred ms historically for VPC attachment), while a warm start adds only ~1–10ms — the gap that drives the entire warm-vs-cold tradeoff.

  • Concurrency limit: a cap (default ~1000/region) protects downstream systems and the platform. Hit it and extra requests are throttled (429 / retried).
  • Scale-up rate: Lambda adds capacity in bursts then ramps; a sudden 0→10,000 spike causes many cold starts at once.
  • Reserved concurrency: carve out a guaranteed slice for a critical function so a noisy neighbour can't starve it.

The execution sandbox & isolation

Multi-tenancy is the cost story: AWS packs many customers' functions onto shared hardware. Security demands that one tenant can't read another's memory or escape the sandbox. AWS built Firecracker, a lightweight micro-VM, to get VM-grade isolation with container-like startup speed (boots in ~125ms) and tiny memory overhead — so they can run thousands of micro-VMs per host densely and safely.

Senior tip: "why not just containers?" Plain containers share the host kernel — a kernel exploit breaks isolation across tenants. Firecracker gives each function its own kernel/VM boundary, which is why it's the right tradeoff for untrusted multi-tenant code.

Tradeoffs & when not to use Lambda

Great fitSpiky/unpredictable load, event-driven glue, infrequent jobs — you pay nothing when idle and scale automatically.
Poor fitSteady high-throughput workloads (a 24/7 server is cheaper), very latency-sensitive paths (cold starts), long-running tasks (15-min cap), heavy local state.

🎯 Interview questions

What is a cold start and why does it happen?

A cold start is when a request has no free pre-initialized environment, so Lambda must create one: start the micro-VM, bootstrap the runtime, and run your init code. It happens on the first request, after scale-up, or after idle environments are reclaimed. Warm starts skip all of this by reusing an existing environment.

Why can global variables persist between invocations, and is that good or bad?

Because Lambda reuses warm environments, module-scope state outlives a single invocation. Good for caching expensive resources like DB connections; dangerous if you store request-specific or user data in globals, since the next request in the same environment could read it.

How does Lambda achieve concurrency, and what happens at the limit?

Each environment serves one request at a time, so concurrency N requires N environments, which Lambda spins up on demand. Past the account/function concurrency limit, additional requests are throttled (429) and event sources may retry.

Why does AWS use Firecracker micro-VMs instead of plain containers?

Plain containers share the host kernel, so a kernel escape breaks tenant isolation. Firecracker gives each function a separate minimal VM (its own kernel) with VM-grade isolation but ~125ms boot and tiny overhead, enabling safe high-density multi-tenancy.

How would you eliminate cold starts for a latency-critical API?

Use Provisioned Concurrency to keep a set of environments pre-initialized and ready, so requests always hit a warm environment. Trade-off: you pay for that reserved capacity even when idle, partially giving up the pay-per-use benefit.

How many concurrent environments do you need for 500 req/sec at 200ms each?

Use concurrency ≈ request rate × average duration = 500 × 0.2s = ~100 environments. That sits well under the default 1,000/region limit, so no throttling — but if duration jumped to 2.5s you'd need ~1,250 and would hit the cap, requiring a limit increase or reserved concurrency.

How Twitter Timeline Works Case Study

Twitter's home timeline is the list of tweets you see from everyone you follow, newest first. The interesting system-design problem is delivery: when someone tweets, how do their tweets reach the timelines of all their followers fast? The answer hinges on a famous tradeoff — fan-out on write vs fan-out on read — and the painful "celebrity" hot-key problem.

Figure: write path fans a tweet id into each follower's Redis timeline cache; reads pull ids from the cache and hydrate from the tweet store. Dashed = celebrity tweets merged in at read time instead of pushed.

Requirements (functional & non-functional)

  • Functional: post a tweet; follow/unfollow users; load your home timeline (merged, reverse-chronological tweets from everyone you follow).
  • Non-functional: extremely read-heavy (timeline loads vastly outnumber tweets); home timeline must load in tens of milliseconds; handle wildly skewed follower counts (most users have hundreds, some have hundreds of millions).
Senior tip: The whole design is driven by one observation: people read their timeline far more often than they tweet. So it's worth doing extra work at write time to make the common operation — reading — almost free.

The core hard problem: fan-out on write vs read

"Fan-out" means spreading one tweet to many timelines. There are two strategies:

Fan-out on write (push)When you tweet, immediately push the tweet id into a precomputed timeline list for every follower. Reading a timeline is then just reading that list — O(1), blazing fast.
Fan-out on read (pull)Store the tweet once. When a user loads their timeline, fetch recent tweets from everyone they follow and merge them on the fly. Cheap writes, expensive reads.

Push makes reads cheap but writes expensive (one tweet = thousands of list inserts). Pull makes writes cheap but reads expensive (one read = merge across many followees). The right answer is usually a hybrid.

The celebrity / hot-key problem

Push falls apart for celebrities. If someone has 100 million followers, a single tweet triggers 100 million list inserts — a "fan-out storm" that can take minutes and hammer the system. That one user's key is a hot key.

Put numbers on the storm. A push costs one Redis list insert (~tens of microseconds), so a normal user with 500 followers triggers ~500 inserts — trivial. A celebrity with 100M followers triggers 100 million inserts for a single tweet. Even at 1M inserts/sec across a sharded cluster that is ~100 seconds of write amplification per tweet, and a few such accounts tweeting together can saturate the fan-out tier — the classic hot key. The break-even threshold (often somewhere around tens of thousands to ~1M followers) is where the cost of pushing exceeds the cost of letting readers pull, which is exactly why those accounts switch strategy.

Gotcha: Pure fan-out-on-write does not scale for celebrities; pure fan-out-on-read does not scale for normal users (every timeline load would merge thousands of accounts). Picking only one strategy is the classic wrong answer.

The hybrid solution

Twitter uses both, chosen per user:

  • Normal users: fan-out on write. Their tweets get pushed into followers' timelines so reads stay O(1).
  • Celebrities: fan-out on read. Their tweets are not pushed. Instead, when you load your timeline, the system merges your precomputed list (from normal followees) with a freshly fetched list of recent tweets from the few celebrities you follow.
Senior tip: This hybrid is a great example of "there is no single right answer — pick per case". You apply the cheap-write strategy exactly where push is too expensive (the long tail of celebrities) and keep the cheap-read strategy everywhere else.

Step-by-step request flow

Trace a tweet from a normal user, then a home-timeline read that also includes a celebrity, using the diagram's components.

  1. Write: a normal user posts via Tweet write. The tweet is saved once to the Tweet store (source of truth) and the write returns immediately; an event is dropped on the Fan-out queue.
  2. Resolve followers: the Fan-out service consumes the event, asks the Social graph for this author's follower ids.
  3. Push (fan-out on write): for each follower it prepends the tweet id to that follower's list in the Timeline cache (Redis), trimming the list to the most recent ~800. Writes are async, so a normal tweet's fan-out finishes within seconds.
  4. Read: a Reader opens their home timeline. The service reads the top ~50 ids straight from that user's precomputed Redis list — an O(1) lookup, no merging across followees.
  5. Celebrity merge (fan-out on read): for the handful of celebrities this user follows, whose tweets were not pushed, the service fetches their recent tweet ids (the dashed path) and merges them by timestamp into the result.
  6. Hydrate: finally the service fetches the actual tweet bodies for just those ~50 ids from the Tweet store (usually cache-backed) and returns the assembled timeline.
Why ids, not tweets, in the cache? The fan-out step writes one tiny id per follower instead of a full tweet object. That keeps the per-follower lists small enough to live entirely in RAM (sub-millisecond reads) and means an edit or delete touches only the single source-of-truth row, not millions of copies.

Data model / storage choice

Tweets (durable store)The tweet content itself, stored once. Source of truth.
Timelines (Redis lists / sorted sets)Per-user precomputed list of tweet ids, capped at the most recent ~800. In-memory so reads are sub-millisecond.
Social graph (followers/following)Who follows whom; drives the fan-out at write time and the celebrity merge at read time.

Why store only tweet ids in the timeline? Ids are tiny, so the in-memory lists stay small and cheap to replicate. The actual tweet content is hydrated from the tweet store (often cached) only for the slice you're about to display.

Scaling and tradeoffs

Timelines are capped (you rarely scroll back thousands of tweets), which bounds memory. Fan-out happens asynchronously through a queue, so tweeting feels instant even while the push to followers is still in flight — accepting brief eventual consistency (a tweet may take a moment to appear) in exchange for fast writes.

Sizing the cache. Suppose ~300M users each keep a capped list of ~800 tweet ids at ~16 bytes each (id + score): that is ~12.8KB/user, so the working set is on the order of a few TB of RAM across a sharded Redis cluster — large but tractable, and the reason the cache stores ids, not multi-KB tweet objects (which would balloon it 50–100x). The payoff is a home-timeline read served in single-digit milliseconds: one Redis range read plus a batched hydrate, against a read load of hundreds of thousands of timeline loads/sec. Tweets themselves arrive far slower (a few thousand/sec), which is precisely why spending extra work at write time to make reads near-free is the correct trade.

Gotcha: Don't store full tweet objects in every follower's timeline list — that multiplies storage by your follower count and makes a tweet edit/delete a nightmare. Store ids and hydrate.

🎯 Interview questions

What is the difference between fan-out on write and fan-out on read?

Fan-out on write (push) copies a tweet id into every follower's precomputed timeline when it's posted, making reads O(1). Fan-out on read (pull) stores the tweet once and merges followees' tweets at read time, making writes cheap but reads expensive.

Why is reading a timeline O(1) in the push model?

Because the work was already done at write time. Each follower has a precomputed list of tweet ids in memory, so loading the timeline is just reading the top of that list — no merging across followees needed.

Why can't celebrities use fan-out on write?

One tweet would fan out to hundreds of millions of timelines — a fan-out storm that's slow and resource-crushing. Their account becomes a hot key. Instead their tweets are pulled (fan-out on read) and merged into followers' timelines at read time.

How does the hybrid approach assemble a user's home timeline?

It merges two sources: the precomputed list pushed by normal followees, plus a freshly fetched set of recent tweets from the small number of celebrities the user follows. The merge is by timestamp, newest first.

Why store only tweet ids in timelines instead of full tweets?

Ids are tiny, keeping the in-memory lists small, cheap to replicate, and easy to cap. The full content lives once in the tweet store and is hydrated (and cached) only for the slice being displayed, which also keeps edits/deletes manageable.

Roughly where is the threshold to switch a user from push to pull, and why?

At the follower count where the write amplification of pushing every tweet (one insert per follower) outweighs the cost of having readers merge the account at read time — often somewhere from tens of thousands up to ~1M followers. Below it, push keeps reads O(1) cheaply; above it, a single tweet would cause a multi-million-insert fan-out storm, so pull is cheaper overall.

Estimate the memory needed for the timeline cache and justify the id-only choice.

With ~300M users each holding ~800 ids at ~16 bytes, that is ~13KB/user, or a few TB of RAM across a sharded Redis cluster. Storing full tweet objects instead would multiply that by ~50–100x and make every edit/delete touch millions of copies, so ids + hydrate-on-read is what keeps the cache affordable and correct.

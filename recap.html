Node.js Core runtime

  • Event loop: single JS thread + libuv thread pool. Phases: timers → pending → poll → check → close.
  • Microtasks drain between phases: nextTick > Promises > then setImmediate (check).
  • Don't block the loop — CPU work → worker_threads; scale a server → cluster/PM2.
  • Streams: Readable/Writable/Duplex/Transform; pipeline() for backpressure + error cleanup.
  • Buffer: raw bytes off the V8 heap; alloc (safe) vs allocUnsafe.
  • Modules: CJS (require, sync, cached singletons) vs ESM (import, async, tree-shakeable).
  • Errors: sync→try/catch; callbacks→(err,data); promises→.catch/await; emitter→'error' event; last resort uncaughtException/unhandledRejection = log & exit.
Is Node single-threaded?

JS runs on one thread; libuv has a background pool (default 4) for fs/dns/crypto/zlib.

setTimeout(0) vs setImmediate?

Inside an I/O callback, immediate fires first (check after poll). Top-level order is non-deterministic.

Express framework

  • Thin layer over http: adds routing, middleware, req/res helpers — no new powers, less boilerplate.
  • Middleware (req,res,next) pipeline — runs in order; must next() or send a response.
  • Types: application (app.use), router (router.use), route-level, built-in, error (4-arg).
  • Error handler = 4 args (err,req,res,next), registered LAST. Express 4 needs an async wrapper; v5 auto-forwards rejections.
  • Router = mini-app for modular routes + scoped middleware/auth.
  • REST: nouns + verbs; 200/201/204, 400/401/403/404/409/422/429, 500/503; PUT/DELETE idempotent.
What does Express add over raw http?

Routing, middleware pipeline, body parsing, req.params/query/body, res.json/redirect, static serving, routers, error handling.

How is an error handler detected?

By arity — exactly four parameters.

NestJS framework

  • Opinionated, TS-first on top of Express/Fastify. Angular-like DI + decorators.
  • 3 blocks: Modules (@Module) · Controllers (@Controller) · Providers (@Injectable).
  • DI container injects via constructor; default scope = singleton; useClass/useValue/useFactory.
  • Module encapsulation: providers private unless exportsed + importsed.
  • Lifecycle order: Middleware → Guards → Interceptors(pre) → Pipes → Handler → Interceptors(post) → Exception Filters.
  • Guards=auth · Pipes=validation · Interceptors=wrap before/after · Filters=errors.
  • Ecosystem: TypeORM/Prisma/Mongoose, ValidationPipe, Passport/JWT, GraphQL, microservices, Swagger, testing.
Guard vs middleware?

Middleware = generic Express-level pre-processing. Guards run later, have ExecutionContext, and exist to allow/deny (auth).

When Nest over Express?

Large/long-lived apps, multiple teams, need for DI, testability, and built-in features. Overkill for tiny APIs.

MongoDB NoSQL

  • Documents/collections; Mongoose = ODM (schemas, validation, populate).
  • Embed vs reference: embed bounded data read together & owned; reference large/shared/unbounded.
  • Indexes: IXSCAN good / COLLSCAN bad; ESR rule (Equality→Sort→Range); leftmost prefix; verify with .explain().
  • Aggregation: $match→$group→$sort→$lookup; match early to use indexes.
  • Atomicity: single-doc always atomic; multi-doc needs transactions (replica set + session).
  • Scale: shard key = high cardinality + even distribution; w:'majority' for durability.
16MB?

Max BSON document size — avoid unbounded embedded arrays (use bucket/subset or reference).

Avoid N+1?

Batch with $in or $lookup instead of populating in a loop.

SQL relational

  • Always pool connections + parameterized queries ($1) to stop SQL injection.
  • Joins: INNER (matches), LEFT (all left + NULLs); WHERE on right table silently makes LEFT act like INNER.
  • ACID: Atomicity, Consistency, Isolation, Durability.
  • Isolation: Read Committed (PG default) → Repeatable Read (MySQL) → Serializable; dirty/non-repeatable/phantom reads.
  • Indexes: B-tree; not used with functions on column, leading %wildcard, type mismatch. Check EXPLAIN ANALYZE.
  • Normalize (1NF–3NF) to kill anomalies; denormalize for read/reporting speed.
  • Locking: pessimistic (FOR UPDATE) vs optimistic (version column).
SQL vs NoSQL?

SQL: relational, ACID, joins, stable schema. NoSQL: flexible, denormalized, horizontal scale. Pick by data shape & access patterns.

PUT vs PATCH?

PUT = full replace (idempotent); PATCH = partial update.

Auth & Security cross-cutting

  • Sessions (stateful, easy revoke, needs store) vs JWT (stateless, scales, hard to revoke).
  • JWT = header.payload.signature; signed not encrypted — never store secrets in payload.
  • Access + refresh tokens; store in httpOnly/Secure/SameSite cookies, not localStorage.
  • Passwords: bcrypt/scrypt/argon2 + per-user salt — never plain SHA.
  • helmet (headers), cors, rate-limit, input validation, timingSafeEqual.
  • Secrets in env vars; npm audit; disable x-powered-by.
Revoke a JWT?

Short-lived access token + revocable refresh token, or a server-side blocklist/version check.

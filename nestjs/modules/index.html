Modules NestJS

A module (@Module) groups related controllers and providers into a cohesive feature unit. Every Nest app has at least a root module (AppModule); larger apps are a tree of feature modules.

The @Module metadata

@Module({
  imports:     [DatabaseModule, AuthModule],  // other modules this one depends on
  controllers: [UsersController],             // request handlers
  providers:   [UsersService],                // injectables, scoped to THIS module
  exports:     [UsersService],                // make providers available to importers
})
export class UsersModule {}

Encapsulation (key concept)

Providers are private to their module by default. If UsersModule wants AuthModule's service, AuthModule must exports it and UsersModule must imports it. This explicit boundary is what keeps large apps decoupled.

Module types

  • Feature module — one per domain (users, orders, billing).
  • Shared module — exports common providers reused across features.
  • Global module (@Global()) — providers available everywhere without importing (use sparingly).
  • Dynamic module — forRoot()/forFeature() pattern for configurable modules (e.g. TypeOrmModule.forRoot(config)).
Senior signal: Mention the dynamic module forRoot/forFeature pattern — it's how config-driven modules (DB, JWT, config) pass options at import time.

🎯 Interview questions

What does a module encapsulate?

Its providers — they're private unless explicitly listed in exports. Consumers must imports the module to use them.

imports vs providers vs exports?

providers: injectables owned by this module. exports: which of those are visible to importers. imports: other modules whose exports you need.

What is a dynamic module?

A module configured at import time via static methods like forRoot(options) / forFeature(...) that return module metadata — used by DB, config, JWT modules.

When use @Global()?

For widely-needed providers (config, logging) to avoid re-importing everywhere. Overuse breaks encapsulation, so keep it minimal.

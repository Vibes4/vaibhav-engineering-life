How Slack Works Case Study

Slack is a team chat app organised into workspaces (a company), channels (rooms inside it), and direct messages. The interesting system-design problem is real-time fan-out: when someone types in #general, every connected member of that channel must see the message in milliseconds — but nobody outside it should. Doing this for millions of concurrent connections is the hard part.

Figure: a message is published once to the pub/sub bus and fanned out by every gateway holding a subscriber, while the durable write goes to the message store.

Requirements

  • Functional: send/receive messages in channels and DMs; create channels; show who is online (presence); show unread badges and counts; full message history that survives reconnects.
  • Non-functional: near real-time delivery (<200ms), highly available, ordered delivery within a channel, must hold millions of long-lived connections, and never lose a message even if the client is briefly offline.

The core hard problem: real-time fan-out

HTTP is request/response — the client asks, the server answers. But chat is server-initiated: the server must push a message to you without you asking. That is why Slack uses WebSockets — a single long-lived, bidirectional TCP connection per client that stays open so the server can push at any time.

Think of a channel as a mailing list. When a message arrives, the server looks up every subscriber of that channel and writes the message down their open socket. This "look up the members, push to each" step is fan-out, and it is the heart of the demo for this page.

Why not just poll?

Short pollingClient asks "anything new?" every few seconds. Simple, but wasteful (mostly empty replies) and laggy (delay = poll interval). Terrible at scale.
Long pollingServer holds the request open until there is news. Better latency, but still one request per message and heavy on connection churn.
WebSocketOne persistent connection, server pushes instantly. Lowest latency and overhead. The standard for chat. Cost: you must manage millions of stateful connections.

Data model and storage

Messages are append-only and read by channel in time order, so the natural model is: messages(channel_id, ts, sender_id, body) partitioned by channel_id and sorted by ts. A wide-column store (Cassandra) or sharded SQL fits this write-heavy, time-ordered access pattern.

  • channel_members: who belongs to each channel — this is the list you fan out to.
  • ts as the ordering key: a monotonic per-channel timestamp/sequence gives a stable order so all clients agree on message order.

Step-by-step request flow

  1. Clients open a single WebSocket to the WebSocket Gateway tier and keep it alive with periodic heartbeats; one gateway box typically holds 100k–1M sockets, so a few hundred boxes cover tens of millions of users.
  2. A user sends "hello" in #general; it travels up the existing socket to whichever gateway hosts that client.
  3. The gateway forwards the message to the Channel/Message service, which assigns a monotonic per-channel sequence number and writes it durably to the Message store first.
  4. Only after the durable write does it publish once to the channel's topic on the Pub/Sub bus.
  5. Every WebSocket Gateway subscribed to that topic receives the single published copy, looks up which of its local sockets belong to #general, and writes the message down each one — that local lookup-and-push is the fan-out.
  6. In parallel, clients ping the Presence service; a missed heartbeat flips a member to "away" with a TTL, never touching the durable store.

Connections, presence and the gateway tier

A user's socket lands on one of many stateless gateway servers. But the sender's gateway may not host the recipient's socket. So gateways subscribe to a shared pub/sub bus (e.g. Redis pub/sub or Kafka): publish the message once to the channel's topic, and every gateway holding a subscriber delivers it locally. This decouples senders from where recipients happen to be connected.

Why this scales: the alternative — a sender's gateway opening a direct connection to every other gateway holding a recipient — is O(N²) in gateway count and re-sends the payload once per gateway. With pub/sub the sender publishes one message regardless of fan-out size, and the bus does the cheap topic-to-subscriber routing. A busy channel with 50k members spread across 300 gateways becomes one publish plus ~300 local socket writes instead of 50k cross-server sends. End-to-end this keeps p99 delivery comfortably under the 200ms target even though each gateway is sustaining hundreds of thousands of idle long-lived TCP connections (the real cost is memory and file descriptors per socket, not CPU).

Presence (online/away) is just a heartbeat: clients ping periodically; miss a few and you are marked away. Store it in a fast in-memory store with a TTL, never the main DB.

Gotcha: Computing unread counts by scanning all messages on every page load melts your database. Instead store a per-user last_read_ts per channel; unread = messages with ts > last_read_ts, and keep a cached running count you increment on new messages.

Delivery guarantees and reconnects

WebSockets drop all the time (wifi, tunnels, sleep). So the socket is for speed, not truth. The database is the source of truth. On reconnect the client sends "last message I saw = ts X" and the server replays everything after X. This is why you persist first, then fan out — never the reverse.

Senior tip: Separate the two paths. (1) Durable path: write to the DB and assign the official sequence number. (2) Fast path: publish to pub/sub for instant delivery. The fast path can lose a message with no harm because the durable path + reconnect replay guarantees eventual delivery and correct order.

🎯 Interview questions

Why WebSockets instead of HTTP requests for chat?

Chat needs the server to push messages without the client asking. Plain HTTP is request/response, so the client would have to poll. A WebSocket is a single persistent bidirectional connection, giving the server a channel to push instantly with minimal overhead.

The sender and receiver are connected to different servers. How does the message reach the receiver?

Gateways are stateless and subscribe to a shared pub/sub bus. The receiving gateway publishes the message to the channel's topic once; every gateway holding a subscriber of that channel gets it and writes it down the right local sockets. Senders never need to know where recipients are connected.

How do you compute unread counts efficiently?

Store a per-user, per-channel last_read_ts. Unread messages are those with ts > last_read_ts. Maintain a cached running count incremented on each new message and reset when the user reads, so you never scan the message history on load.

A user's connection dropped for 30 seconds. How do they not miss messages?

The DB is the source of truth and assigns ordered sequence numbers. On reconnect the client reports the last ts it saw and the server replays everything after it. The real-time push is just an optimisation on top of this durable log.

How do you ensure everyone sees messages in the same order?

Assign a monotonic per-channel sequence number/timestamp at write time in the durable path, and have all clients order by it. Even if the fast pub/sub path delivers slightly out of order, clients sort by the authoritative sequence so they converge on one order.

Why use a pub/sub bus instead of gateways talking to each other directly?

Direct gateway-to-gateway delivery is O(N²) in the number of gateways and re-sends the payload once per gateway holding a subscriber. With a pub/sub bus the sender publishes exactly one message to the channel topic regardless of audience size, and the bus handles topic-to-subscriber routing. A 50k-member channel becomes one publish plus a few hundred local socket writes instead of tens of thousands of cross-server sends.

What actually limits how many connections one gateway can hold?

Chat connections are mostly idle, so CPU is rarely the bottleneck — the limits are memory per socket (buffers, TLS state) and OS file descriptors. You tune the kernel (fd limits, socket buffers) and run many stateless gateways behind the pub/sub bus, scaling horizontally to 100k–1M sockets per box.

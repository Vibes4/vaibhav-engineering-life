NestJS Overview NestJS

NestJS is an opinionated, TypeScript-first framework for building scalable server-side apps. It runs on top of Express (or optionally Fastify) and adds structure inspired by Angular: modules, controllers, providers, heavy use of decorators and dependency injection.

The three building blocks

  • Modules (@Module) — organize the app into cohesive feature units; the root module bootstraps everything.
  • Controllers (@Controller) — handle incoming requests and return responses (the routing layer).
  • Providers / Services (@Injectable) — business logic, injected via DI.

Why NestJS over plain Express?

  • Structure out of the box — Express is unopinionated; large Express apps drift into inconsistent folder layouts. Nest enforces a modular architecture.
  • Dependency Injection — built-in IoC container → loosely coupled, highly testable code.
  • TypeScript + decorators — strong typing, declarative routing/validation.
  • Batteries included — validation pipes, guards (auth), interceptors, exception filters, config, testing utilities, Swagger, microservices, GraphQL, WebSockets.
  • Platform-agnostic — swap Express for Fastify with one line for more throughput.
One-liner: "Nest is to Express what Angular is to vanilla JS — an opinionated, DI-driven, modular layer that trades some flexibility for structure, testability, and scale."
Trade-off: More boilerplate and a steeper learning curve (decorators, DI, modules). Overkill for a tiny API; excellent for large teams and long-lived enterprise apps.

Request flow at a glance

Request → Middleware → Guards → Interceptors(before) → Pipes → Controller → Service → Interceptors(after) → Response (Exception Filters catch any error along the way).

🎯 Interview questions

Is NestJS a replacement for Express?

No — it sits on top of Express (default HTTP adapter) or Fastify. It adds architecture, DI, and features; the underlying HTTP server is still Express.

What are the three core building blocks?

Modules (organization), Controllers (routing), Providers/Services (business logic injected via DI).

When would you choose Nest over Express?

Large/long-lived apps, multiple teams, need for testability and consistency, or when you want built-in validation/auth/DI. For a tiny service, plain Express/Fastify is simpler.

Express vs Fastify adapter in Nest?

Nest is platform-agnostic; you can run on @nestjs/platform-express (default) or @nestjs/platform-fastify for higher throughput, with the same app code.

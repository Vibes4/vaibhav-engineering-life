Providers & Dependency Injection NestJS

DI is the heart of NestJS. A provider is any class marked @Injectable() that the Nest IoC container creates and injects where needed — you never new a service yourself.

How injection works

@Injectable()
export class UsersService {
  constructor(private readonly db: DatabaseService) {}   // db injected automatically
  findOne(id: string) { return this.db.query('users', id); }
}

@Controller('users')
export class UsersController {
  constructor(private readonly users: UsersService) {}   // service injected automatically
}

Nest reads the constructor parameter types (via TypeScript metadata), looks them up in the container, and supplies a single shared instance.

Why DI matters

  • Loose coupling — depend on an abstraction, not a concrete new.
  • Testability — swap real providers for mocks in unit tests (Test.createTestingModule + .overrideProvider()).
  • Singletons by default — one instance per app, reused everywhere (efficient).

Custom & advanced providers

  • useClass — bind a token to an implementation (swap by environment).
  • useValue — inject a constant/mock.
  • useFactory — compute the provider (with its own injected deps) — e.g. a configured client.
  • Injection tokens — inject interfaces/non-class values via a string/Symbol token + @Inject(TOKEN).

Provider scopes

  • DEFAULT (singleton) — one instance shared app-wide.
  • REQUEST — a new instance per request (carries per-request state; slower).
  • TRANSIENT — a fresh instance for every consumer.
Circular dependencies: two providers needing each other → use forwardRef(() => X), but treat it as a smell to refactor.

🎯 Interview questions

What is dependency injection here?

The IoC container instantiates providers and injects them via the constructor based on types/tokens — you declare what you need, Nest supplies it.

Default provider scope?

Singleton — one instance per application, shared across the app. REQUEST and TRANSIENT scopes exist for per-request/per-consumer instances.

useClass vs useValue vs useFactory?

useClass binds a token to an implementation; useValue injects a constant/mock; useFactory computes the provider (can inject deps) — e.g. configured clients.

How does DI help testing?

Replace real dependencies with mocks via the testing module's overrideProvider, so units are isolated.

How do you inject something that isn't a class (e.g. config)?

Use an injection token (string/Symbol) with a custom provider and @Inject(TOKEN).

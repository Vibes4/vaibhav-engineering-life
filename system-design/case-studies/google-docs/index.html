How Google Docs Works Case Study

Google Docs lets many people edit the same document at the same time and somehow everyone ends up seeing the identical text. The interesting problem is conflict resolution: if two people type into the same place at the same instant, whose change wins, and how do all replicas converge to one final document without anyone's keystroke getting lost?

Figure: every client sends ops tagged with a base revision; the collaboration server transforms, sequences, appends to the op log, and broadcasts the transformed op back.

Requirements

  • Functional: multiple users edit one document live; everyone sees each other's changes within ~100ms; cursors and selections are shared; full edit history / undo.
  • Non-functional: convergence (all replicas end identical), intention preservation (your edit means the same after others' edits applied), low latency, works offline then merges on reconnect.

Why naive last-write-wins fails

Imagine the doc is "cat". Alice inserts "s" at the end to make "cats"; at the same moment Bob inserts "!" at the end. If we just send raw positions, Bob's "insert at position 3" applied after Alice's insert lands in the wrong spot, and the two replicas disagree. Sending the whole document on every keystroke "fixes" it but is unusable at scale and clobbers concurrent edits. We need edits to be transformed against each other.

The core idea: Operational Transformation (OT)

OT models every edit as an operation like insert("s", pos 3) or delete(pos 0). When two operations are made concurrently against the same base, before applying the second one you transform it against the first so its position accounts for what the first one already did. Then applying them in either order yields the same result. That property is what the demo proves.

  • transform(a, b): rewrites operation b so it still expresses the user's intent after a has been applied.
  • Example: Alice inserts at index 1, Bob inserts at index 4. Bob's index must shift to 5, because Alice's insert pushed everything after index 1 one slot to the right.

OT vs CRDT

Operational Transformation (OT)What Google Docs uses. Compact operations, but needs a central server to order and transform ops. Transform functions are notoriously tricky to get correct for every op pair.
CRDT (Conflict-free Replicated Data Type)Each character gets a unique, globally-sortable id; merging is automatic and needs no central authority, so it is great for peer-to-peer/offline. Cost: more metadata per character and tombstones for deletes.

Step-by-step request flow

  1. Client A (Alice) types "s"; her editor emits insert("s", pos 3) tagged with the base revision she last saw (say rev 41) and sends it over the WebSocket to the Collaboration server.
  2. Concurrently Client B (Bob) emits insert("!", pos 3) also based on rev 41 and sends it over his own WebSocket.
  3. The Collaboration server (the OT transform engine + sequencer) accepts Alice's op first, assigns it rev 42, and appends it to the Op log.
  4. Bob's op arrives based on rev 41, but the server is now at rev 42, so it transforms Bob's op against Alice's accepted op (shifting his index), assigns it rev 43, and appends that to the Op log.
  5. The server broadcasts each transformed op plus its new revision back over every open WebSocket; Alice receives Bob's transformed op, Bob receives Alice's, and both editors apply them to converge on identical text.
  6. Periodically the server folds the op log into a compacted snapshot in the Document store so a new joiner loads one snapshot plus a short tail of ops instead of replaying the entire history.

Architecture: the central sequencer

OT relies on a server that assigns each operation a global revision number. A client sends an op tagged with the revision it was based on. If the server has since accepted other ops, it transforms the incoming op against them, applies it, broadcasts the transformed op plus the new revision, and clients reconcile. This turns a chaotic free-for-all into a single agreed sequence of operations.

Scale and the WHY: the target is sub-100ms echo of remote edits, and a single human types at most ~5–10 ops/sec, so even a doc with 50 simultaneous editors is only a few hundred ops/sec — trivial throughput. The hard part is not volume but correctness and serialization: every op for one document must funnel through a single authority so revisions are assigned in a strict total order. That is why each document is pinned to one collaboration-server instance (sharded by doc id); spreading one doc's ops across servers would reintroduce the very ordering ambiguity OT exists to remove. The op log is an append-only sequence, so it is cheap to write and to replay; the document store snapshot exists purely to bound replay cost — without it, opening a year-old doc would mean transforming millions of historical ops.

Gotcha: Sending absolute cursor positions without transforming them is the classic bug. Concurrent inserts before your position silently shift everything, so your "insert at 5" lands in the wrong place and replicas diverge. Always transform positions against concurrent ops.

Offline editing and convergence

When a client goes offline it keeps queuing operations against its last known revision. On reconnect it replays them, transforming each against everything the server accepted while it was away. Because OT guarantees convergence, the offline edits slot into the right places and everyone ends with the same text.

Senior tip: The litmus test for any collaborative algorithm is: do two concurrent ops applied in opposite orders produce the same document? If yes you have convergence. The demo runs Alice-then-Bob and Bob-then-Alice and shows both reach the identical string.

🎯 Interview questions

Why can't you just use last-write-wins for a shared document?

Last-write-wins discards one user's edit entirely, and raw positions become invalid once a concurrent edit shifts the text. You lose keystrokes and replicas diverge. Collaborative editing needs every concurrent edit preserved and repositioned, which is what OT/CRDT provide.

What does the transform() function do in OT?

It rewrites one operation so it still expresses the user's intent after another concurrent operation has been applied. Typically it shifts an insert/delete index to account for characters another op added or removed before that position, so applying both ops in any order converges.

OT vs CRDT -- when would you pick each?

OT is compact but needs a central server to order and transform operations; it suits server-coordinated apps like Google Docs. CRDTs merge automatically without a central authority, ideal for peer-to-peer and offline-first apps, at the cost of extra per-character metadata and tombstones.

How are concurrent edits ordered consistently across clients?

A central sequencer assigns each accepted operation a global revision number. Incoming ops are transformed against any ops accepted since their base revision, applied, then broadcast with the new revision. Every client converges on this single agreed sequence.

How does offline editing reconcile when the user comes back online?

The client queues operations against its last-known revision while offline. On reconnect it replays them, transforming each against everything the server accepted in the meantime. OT's convergence guarantee ensures the merged result matches every other replica.

Why pin a whole document to a single collaboration-server instance?

OT needs one authority to assign revisions in a strict total order. If two servers accepted ops for the same doc independently you would reintroduce ordering ambiguity. Throughput is tiny (humans emit only a handful of ops/sec), so a single instance per doc — sharded by document id — easily handles even dozens of concurrent editors while guaranteeing serialization.

Why keep both an op log and a document store?

The op log is the authoritative append-only sequence of revisions, cheap to write and replay and used for transforms, history, and undo. The document store holds periodic compacted snapshots so opening an old document loads one snapshot plus a short tail of ops, bounding replay cost instead of transforming the entire history from scratch.

How a Payment System Works Case Study

A payment system moves money between accounts and external processors. The fascinating thing is that, unlike most web systems, it optimizes for correctness over availability: it is far better to reject a payment than to charge someone twice or lose a cent. This forces three classic patterns — idempotency, the double-entry ledger, and the saga — that every senior engineer must know cold.

Figure: the API dedupes on the idempotency store, the ledger posts double-entry rows, the processor captures the money, and a dashed async webhook later confirms the final state.

Step-by-step request flow

  1. Client sends "charge $50" with a unique idempotency key it reuses on every retry.
  2. Payment API looks the key up in the Idempotency Store. A hit returns the stored result immediately (no double charge); a miss proceeds.
  3. Ledger Service (the star) posts the double-entry rows — a debit and a matching credit — in one atomic transaction together with the idempotency key, moving the entry into a PENDING state.
  4. Payment Processor is called to actually capture the funds on the card network or bank.
  5. Webhook (async, dashed) arrives from the processor later, and the ledger transitions the entry to SETTLED or REVERSED. A nightly reconciliation job backstops any missed webhook.

Requirements: correctness is non-negotiable

  • Functional: charge/refund, transfer between accounts, talk to external processors (Stripe, banks), record every movement, report balances.
  • Non-functional: exactly-once money movement (the holy grail), auditability (every cent traceable), strong consistency of balances, durability, and reasonable availability — but availability bends before correctness ever does.
Senior tip: Phrase it as the CAP tradeoff: a payment system chooses C over A. When in doubt, it stops and asks/retries rather than guessing — because an incorrect debit is a real-world financial loss, while a brief outage is merely an inconvenience.

The core hard problem: never charge twice

Networks are unreliable. A client sends "charge $50", the charge succeeds, but the response is lost. The client retries. Without protection, the user is charged $100. This is the at-least-once delivery reality of distributed systems — you cannot prevent retries, so you must make them safe.

The fix is the idempotency key: the client generates a unique key per logical operation and sends it with every retry. The server records the key with the result. If a request arrives with a key it has already processed, it returns the stored original result instead of doing the work again.

Gotcha: The idempotency record and the actual money movement must be committed in the same atomic transaction. If you write the ledger entry but crash before saving the key, a retry double-charges. If you save the key but crash before the ledger entry, the money never moves but retries are blocked. Atomicity binds them together.

Data model: the double-entry ledger

Banks have used double-entry bookkeeping for 500 years because it makes errors mathematically visible. Every transaction writes at least two entries: a debit from one account and a matching credit to another. The iron law: total debits always equal total credits. Money is never created or destroyed, only moved.

Mutable balance columnSimple, but you only see the current number. No history, hard to audit, and a bug can silently make money appear/vanish with nothing to detect it.
Append-only double-entry ledgerBalance is the sum of immutable entries. Full audit trail, and the debits==credits invariant is a built-in self-check. The industry standard.
  • Append-only: entries are never updated or deleted. A correction is a new reversing entry, preserving history.
  • Balance = sum of entries: derived, never stored as a mutable truth, so it can always be recomputed and verified.

Talking to the outside world: the saga pattern

A real payment spans multiple systems — your ledger, a fraud check, the card network, the merchant's bank. You cannot hold one giant ACID transaction across all of them (they're separate companies with separate databases). Instead you use a saga: a sequence of local transactions, each with a compensating action that undoes it if a later step fails.

  • Step: reserve funds → call processor → confirm. Each step commits locally.
  • Compensation: if the processor declines, run "release reserved funds" to roll back logically — you can't un-commit, so you post a reversing entry.
Senior tip: Sagas trade atomicity for eventual consistency. There will be brief windows where money is "in flight" (reserved but not settled). Model these explicitly as states (PENDING, SETTLED, REVERSED) rather than pretending the transfer is instantaneous — hiding the in-flight state is how bugs hide too.

Consistency with external processors

The external processor (e.g. a card network) is the source of truth for whether money actually moved, and it's eventually consistent. Two tools keep your ledger aligned:

  • Webhooks: the processor calls you back when a charge settles or fails, so you update the in-flight entry to its final state.
  • Reconciliation: a daily batch job compares your ledger against the processor's report and flags any mismatch — the ultimate safety net that catches lost webhooks or bugs.

Putting numbers on it: webhooks are best-effort, not guaranteed — processors typically retry a failed delivery with exponential backoff for up to ~72 hours, and a small fraction never arrive. That is precisely why reconciliation is non-negotiable rather than a nice-to-have. On scale: the ledger must hold the line on strong consistency for balances (a read of an account must reflect every committed entry), while the surrounding API can run at, say, p99 < 300 ms for the synchronous authorize step. Settlement itself is deliberately asynchronous and can take T+1 to T+2 days for card rails — so the system models money as in-flight (PENDING) for hours or days, never pretending the transfer is instant. The reconciliation invariant is exact: sum of debits == sum of credits, checked across every account, every night; a single mismatched cent is treated as a sev-1 incident.

🎯 Interview questions

What is an idempotency key and how does it prevent double charges?

A unique client-generated identifier for one logical operation, sent with the original request and every retry. The server stores it alongside the operation's result. On seeing a key it has already processed, it returns the saved result instead of repeating the work. This turns an unsafe at-least-once retry into a safe exactly-once effect.

Why must the idempotency key be saved in the same transaction as the money movement?

So they succeed or fail together. If only the money moves and the key isn't saved, a retry re-charges; if only the key is saved, the money never moves but retries are blocked. A single atomic transaction guarantees the key exists exactly when the effect exists, making the operation truly idempotent across crashes.

What is double-entry bookkeeping and why use it in software?

Every transaction records equal debits and credits across accounts, so total debits always equal total credits. In software it gives an append-only audit trail and a built-in invariant: if debits != credits, you have a bug. It makes money impossible to silently create or destroy, which is exactly the guarantee a payment system needs.

What is the saga pattern and when do you need it?

A way to coordinate a transaction across multiple independent systems that can't share one ACID transaction. It's a sequence of local transactions, each paired with a compensating action that logically undoes it. If a later step fails, you run the compensations in reverse. You need it whenever a business operation spans services/databases you don't control jointly.

How do you stay consistent with an external processor that's eventually consistent?

Model in-flight states (PENDING/SETTLED/REVERSED) explicitly, react to the processor's webhooks to move entries to their final state, and run a periodic reconciliation job that diffs your ledger against the processor's report to catch any missed updates. Reconciliation is the backstop that guarantees the books eventually agree.

If webhooks can be lost, why not just rely on them and skip reconciliation?

Webhooks are best-effort delivery over the public internet — they can be dropped, delayed, duplicated, or arrive out of order, and processors only retry for a bounded window (often ~72h). Relying on them alone means a lost webhook leaves an entry stuck in PENDING forever, silently diverging your books from reality. Reconciliation is the authoritative, pull-based source of truth: by diffing your ledger against the processor's settlement report on a schedule, you catch every gap regardless of webhook reliability. Webhooks make you fast; reconciliation makes you correct.

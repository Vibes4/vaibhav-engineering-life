How Tinder Works Case Study

Tinder is a location-based dating app: it shows you a deck of nearby people, you swipe right (interested) or left (pass), and when two people swipe right on each other, it's a match and they can chat. The interesting system-design problems hiding behind that simple UI are geo-proximity search (find people near me, fast, at planet scale) and mutual-match detection (a write on my side has to be checked against a write on your side).

Figure: a swipe enters via the API, candidates come from the geo index + recommendation service, the swipe is stored, and the match service does an O(1) reverse-lookup that fires a push on a mutual right-swipe.

Requirements (functional & non-functional)

Before designing anything, pin down what the system must actually do. Mixing up "must-have" and "nice-to-have" is the classic way to over-engineer.

  • Functional: store user profiles; given my location + preferences, return a deck of nearby candidates; record swipes; detect a mutual match and notify both users; let matched users chat.
  • Non-functional: low latency for the deck (the app must feel instant, <200ms); very high read volume (people swipe a LOT — billions of swipes/day); eventual consistency is fine for the deck, but match detection should feel real-time.
Senior tip: Tinder is a read-heavy, write-heavy, but not strongly-consistent system. Nobody is harmed if you briefly see a profile that just went offline. Recognising where you can relax consistency is what lets the system scale.

Step-by-step request flow

Following the components in the diagram above, here is what happens from a right-swipe to a match notification:

  1. Client → API gateway: the app sends the swipe event (and, when the deck runs low, a request for more candidates). The gateway authenticates the user and routes the call.
  2. API → Geo index: to build a deck, the service converts your (lat, lng) to a geohash prefix and range-scans the matching, region-sharded cells to gather a few thousand nearby user IDs.
  3. API → Recommendation service: those candidates are filtered by age/gender/preferences, de-duplicated against your "seen" set, ranked, and returned as a batch so you can swipe several cards without a round-trip per card.
  4. API → Swipe store: each swipe is written as a directed key likes:A:B in a fast KV store. Left swipes can be fire-and-forget.
  5. Swipe store → Match service (★): on a right swipe, the match service does a single O(1) reverse-lookup of likes:B:A.
  6. Match service → Push notification: if the reverse edge exists, both edges now point at each other, so a match event fans out a push to both devices and opens a chat thread.

The core hard problem: "find people near me"

Naively, "find users within 10km of (lat, lng)" means scanning every user on Earth and computing distance. That's O(all users) per request — impossible at scale. The trick is to turn 2D geography into a 1D sortable key so a database index can do the work.

  • Geohash: recursively divide the world map into a grid of cells, each labelled with a short string (e.g. 9q8yy). Nearby points share a prefix. So "find people near me" becomes "find users whose geohash starts with 9q8yy" — a cheap prefix/range scan on an indexed column.
  • Geosharding: partition users across servers by region. A request only ever hits the few shards covering your area, not the whole fleet. This is how you narrow billions of users down to a few thousand candidates before any expensive filtering.
Geohash / quadtreeSimple prefix lookups, indexable in any DB, easy to shard. Cells have fixed sizes, so a dense city packs millions into one cell — needs sub-division.
S2 cells (Google)More uniform cell areas on a sphere, better for global accuracy. More complex to reason about and implement.
PostGIS / native geo index (R-tree)Exact distance queries, rich geo operators. Harder to shard horizontally across regions at extreme scale.

The swipe pipeline

A swipe is a tiny but enormously frequent event. The flow:

  • 1. Fetch deck: client asks the recommendation service; it uses your geohash to pull nearby candidates, filters by age/gender/preferences, ranks them, and returns a batch (so you can keep swiping offline without a round-trip per card).
  • 2. Record swipe: each swipe writes (swiper, swipee, direction) to a fast key-value store. Left swipes can even be fire-and-forget.
  • 3. Check for match: on a right swipe, the service checks "has the other person already swiped right on me?" If yes → match.
Gotcha: don't re-show profiles the user already swiped on. You need a fast "have I seen this person?" check — often a Bloom filter or a per-user seen-set in Redis — applied while building the deck, or the experience feels broken.

Detecting a mutual match

This is the heart of the demo. Store each right-swipe as a directed edge. A match exists when both directed edges exist:

  • A swipes right on B → store key likes:A:B = true.
  • On that write, look up the reverse key likes:B:A. If it exists, both like each other → emit a match event.

Because the check happens at write time on a single key lookup, match detection is O(1) and feels instant — no scanning, no batch job required. The match event then fans out a push notification to both devices.

Senior tip: model "who likes whom" as a directed graph and a match as a bidirectional edge. Doing the reverse-lookup synchronously on the right-swipe write is what makes matches real-time without a separate matching job.

Data model & storage choices

Different sub-systems have different access patterns, so they use different stores — a key senior insight is that one database rarely fits everything.

Swipes / likesKey-value store (e.g. DynamoDB/Redis) keyed by swiper:swipee. Huge write volume, simple point lookups for match detection.
Geo indexGeohash column on an indexed/sharded store for fast prefix range scans by location.
ProfilesDocument store — flexible schema (bios, photos, prompts) read on deck-build.
Matches & messagesStored with a chat service; matches list is read often, messages need ordering.

Scaling & tradeoffs

The deck is read far more than anything is written, so cache aggressively. Pre-compute candidate lists per geohash cell and refresh them periodically rather than recomputing on every request. Hot cities (cells with millions of active users) need to be sub-divided into smaller geohash cells so no single shard is overwhelmed.

To put real numbers on it: Tinder has reported on the order of 1.6+ billion swipes per day (~18k+/sec average, with peaks several times higher in the evening "prime swipe" window). Reads dominate — building a deck touches profiles, the geo index, and the seen-set, so a single deck request can fan out to dozens of point lookups, while a swipe is one tiny KV write. The deck target is <200ms p99 end-to-end, which is only achievable if the geohash range scan returns a pre-narrowed candidate set (a few thousand, not billions) and the rest is served from cache. Why pre-compute? Recomputing a ranked deck on every request would multiply that fan-out by the swipe rate; refreshing a per-cell candidate list every few minutes instead trades a little freshness for one to two orders of magnitude less read work per swipe.

  • Stale decks: trading freshness for speed means you might briefly recommend someone who just left town — acceptable.
  • Asymmetric load: a few extremely popular profiles get a disproportionate share of right-swipes (celebrity problem); rate-limit and fan-out carefully.

🎯 Interview questions

Why use a geohash instead of computing distance directly?

Computing distance to every user is O(N) and can't use an index. A geohash maps 2D location to a 1D string where nearby points share a prefix, so "near me" becomes an indexed prefix/range scan that returns a small candidate set in milliseconds.

How do you detect a mutual match efficiently?

Store each right-swipe as a directed key likes:A:B. On the write, do a single O(1) lookup of the reverse key likes:B:A. If it exists, both like each other → emit a match event. No scanning or batch job needed.

How do you avoid showing the same profile twice?

Keep a per-user "seen" set (Redis set or a Bloom filter) and filter it out while building the deck. A Bloom filter is space-efficient and fast, at the cost of rare false positives (occasionally hiding someone you haven't seen).

What happens in a very dense city where one geohash cell holds millions of users?

Fixed-size cells become hotspots. You sub-divide hot cells into finer geohash precision (longer prefix) so each shard handles a manageable population, and you load-balance those sub-cells across servers.

Where is it safe to relax consistency, and where is it not?

The deck can be eventually consistent — briefly stale recommendations are harmless. Match detection and the resulting notification should feel real-time, so the reverse-lookup is done synchronously on the swipe write rather than in a delayed batch.

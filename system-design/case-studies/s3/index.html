How AWS S3 Works Case Study

S3 stores trillions of objects and serves millions of requests per second, yet the interface is dead simple: PUT key, GET key. The interesting system-design problem is everything hidden behind that simplicity — how do you spread trillions of objects across thousands of machines so that any key is found instantly, no single machine is a bottleneck, data survives disk and datacentre failures, and you can add capacity without re-shuffling everything?

Figure: the consistent-hash ring decides which nodes hold a key's replicas/shards; metadata records the mapping.

Requirements (functional & non-functional)

  • Functional: store an object under a key, retrieve it by key, delete it, list keys by prefix. Objects range from a few bytes to terabytes.
  • Non-functional: effectively unlimited scale, extreme durability (S3 advertises 99.999999999% — eleven nines), high availability, low latency, and the ability to grow the cluster without downtime.
Senior tip: "Eleven nines of durability" is the headline non-functional requirement. It is achieved not by trusting any one disk but by storing redundant copies (or erasure-coded fragments) across many independent failure domains, so multiple things must fail simultaneously to lose data.

Step-by-step request flow

  1. A Client issues PUT key (or GET key) over HTTP.
  2. The API / Load Balancer is the stateless front door; it spreads requests across many identical request-routing servers.
  3. The Consistent-hash ring router (the star) hashes the key onto the ring and walks clockwise to find the node(s) that own it — using virtual nodes so load spreads evenly.
  4. The object is written to several consecutive Storage nodes as full replicas or erasure-coded shards, placed in different racks and availability zones.
  5. The Metadata / index service records the key → placement mapping (and supports prefix listing), so later GETs and LISTs resolve quickly.

A flat key namespace, not a filesystem

S3 looks like folders, but that is an illusion. There are no directories — just a giant flat map from a key string to an object. The slashes in photos/2024/cat.jpg are ordinary characters in the key; "listing a folder" is really "list keys with this prefix". This matters enormously: a real filesystem tree has hot directory nodes that become bottlenecks and limit how wide you can scale. A flat key space has no such central node — keys can be distributed by hashing them, and any key is independent of any other.

Gotcha: Treating S3 like a filesystem leads to bad assumptions — e.g. expecting "rename folder" to be cheap. It isn't; there are no folders to rename, so it means copying every matching object under a new key. Designing key names well (and understanding listing is a prefix scan) avoids painful surprises.

The core hard problem: which machine holds this key?

With thousands of storage nodes, you need a function key -> node. The naive choice is node = hash(key) % N where N is the number of nodes. It works perfectly... until you add or remove a node. Then N changes, and almost every key remaps to a different node. For S3 that would mean shuffling trillions of objects across the network whenever you add a server. Unacceptable.

Gotcha: hash(key) % N is the classic trap. Changing N from 100 to 101 doesn't move ~1% of keys — it moves roughly 99% of them, because the modulus of nearly every hash changes. Never use plain modulo hashing for a cluster whose size will change.

The fix: consistent hashing

Consistent hashing places both nodes and keys on the same circular hash space (a "ring" from 0 to some max, wrapping around). To find a key's home, you hash the key to a point on the ring and walk clockwise to the next node. The beauty: when you add a node, it only steals the keys in the arc between it and its predecessor — roughly 1/N of all keys move, and only from one neighbour. Every other key stays put.

Senior tip: Mention virtual nodes. Placing each physical server at many points on the ring smooths out the otherwise lumpy arc sizes and means that when a node leaves, its load is redistributed across many neighbours instead of dumped on one.

Durability via replication & erasure coding

Once a key maps to a position on the ring, S3 stores it on not one but several consecutive nodes — ideally in different racks and availability zones so a single failure can't take out all copies.

Replication (N copies)Store 3 full copies. Simple, fast reads (read any copy), but uses 3x the storage.
Erasure codingSplit the object into k data fragments + m parity fragments; any k of the (k+m) fragments can rebuild it. Far less storage overhead for the same durability — the standard for large objects.

Put numbers on the tradeoff. Three full replicas survive 2 simultaneous failures but cost 3x the raw storage (200% overhead). A typical erasure code like 10+4 (10 data + 4 parity fragments) survives any 4 of 14 fragments being lost yet costs only 1.4x storage — a 60%+ saving for comparable or better fault tolerance. That is why S3 erasure-codes large objects. The reason this all adds up to eleven nines (99.999999999% — losing one object in 100 billion per year) is independence: fragments are spread across separate disks, racks, and at least three availability zones, so the failures must be statistically independent and simultaneous. A continuous background process also scrubs and re-replicates fragments the moment a disk dies, so the system is constantly racing back to full redundancy rather than waiting for the next failure.

Consistency model

S3 now offers strong read-after-write consistency: once a PUT returns success, every subsequent GET sees the new data. Historically it was eventually consistent for overwrites, which surprised many engineers whose code read a key immediately after writing and got the old value. The lesson: always know your store's consistency guarantees, because they directly shape what your application code can safely assume.

Gotcha: Even with strong consistency, listing (the prefix scan) and replication across regions can still lag. "Read-after-write on a single object key" is not the same as "the whole bucket index is instantly globally consistent". Don't over-generalise the guarantee.

🎯 Interview questions

Why is S3's key namespace flat instead of a directory tree?

A tree has central directory nodes that become hotspots and limit horizontal scale. A flat key-to-object map has no central node, so keys can be hashed and spread across thousands of machines independently. Folders are just an illusion created by slashes in key names; listing a folder is a prefix scan.

What's wrong with using hash(key) % N to pick a storage node?

When N changes (you add or remove a node), the modulus of nearly every key changes, so roughly all keys remap and must be moved. For a cluster storing trillions of objects, that means shuffling almost everything on every scaling event — completely impractical.

How does consistent hashing fix that?

Nodes and keys sit on the same circular hash ring; a key belongs to the next node clockwise. Adding a node only captures the arc between it and its predecessor — about 1/N of keys, taken from one neighbour. All other keys stay where they are, so scaling moves minimal data.

What are virtual nodes and why do they matter?

Each physical server is placed at many points on the ring instead of one. This evens out the otherwise random, lumpy arc sizes so load is balanced, and when a node leaves its keys are spread across many neighbours rather than dumped entirely on one successor.

How does S3 achieve eleven nines of durability?

It never trusts one disk. Each object is redundantly stored across many independent failure domains (different disks, racks, availability zones) via replication or erasure coding, where any k of k+m fragments can rebuild the object. Losing data requires many independent failures at once.

How Apache Kafka Works Case Study

Kafka looks like a message queue, but at heart it is something simpler and more powerful: a distributed, durable, append-only commit log. Producers write records to the end of a log; consumers read forward through it at their own pace. The interesting system-design problem is how to make that log scale horizontally, survive machine failures, preserve ordering where it matters, and let many independent consumers read the same data without stepping on each other.

Figure: each partition is an independent ordered log; one consumer per partition within a group.

Requirements (functional & non-functional)

  • Functional: producers publish records to a named topic; consumers subscribe and read records; consumers can replay from any past position; multiple independent consumer applications can read the same topic.
  • Non-functional: very high throughput (millions of records/sec), durability (a written record is not lost if a broker dies), horizontal scalability, and ordering guarantees at least within a related stream of events.
Senior tip: The mental unlock is "Kafka is a log, not a queue". In a traditional queue, reading a message removes it. In Kafka, reading does not delete anything — the consumer just advances a bookmark (an offset). That one difference is why replay, multiple consumers, and durability all become easy.

Step-by-step request flow

  1. Producers publish a record, choosing a partition by hashing the record's key (or round-robin if keyless).
  2. The Broker receives the record into the leader of the target partition (P0/P1/P2) — the partitioned commit log that is the heart of Kafka.
  3. The leader appends the record to its log, assigning the next offset, and the replicas copy it. With acks=all the write is acknowledged only once every in-sync replica has it.
  4. Each consumer in the Consumer group (C1, C2) is assigned a disjoint set of partitions, so no two consumers in the group read the same partition.
  5. Consumers poll their partitions, process records in offset order, and periodically commit their position back to Kafka so a restart resumes cleanly.

The core abstraction: an append-only log

A log is the simplest possible storage structure: you can only add to the end, and every record gets a monotonically increasing number (its offset) — 0, 1, 2, 3.... Appending is just a sequential disk write, which is astonishingly fast (disks are great at sequential I/O, terrible at random). Because records are never modified in place, there are no locks, no in-place updates, and reads never block writes.

Gotcha: Freshers often assume Kafka deletes a message once it's consumed. It does not. Records live until a retention policy (time- or size-based) expires them, completely independent of whether anyone has read them. This is exactly what lets a brand-new consumer start reading from offset 0 — last week's data is still there.

Partitions: how the log scales

A single log on a single machine has a ceiling — one disk, one CPU. So Kafka splits each topic into partitions, and each partition is an independent log that can live on a different broker (server). Throughput now scales with partition count: 10 partitions = 10 logs being written in parallel.

But there's a cost. Ordering is guaranteed only within a single partition, never across the whole topic. Offsets are per-partition. So how do you keep related events in order? You route them to the same partition using a key.

Partition by key (hash the key)All records with the same key (e.g. user_42) hash to the same partition, so that user's events stay strictly ordered. This is the common choice when order-per-entity matters.
Round-robin (no key)Records spread evenly across partitions for maximum parallelism, but you lose any ordering guarantee between them. Fine for independent events like raw metrics.

Consumer groups & offsets

A consumer group is how Kafka shares work. Within one group, each partition is read by exactly one consumer, so adding consumers (up to the partition count) splits the load. Crucially, different groups are completely independent — the same topic can feed a billing service and an analytics service, each tracking its own position.

That position is the committed offset: "consumer group G has processed partition P up to offset N". The consumer periodically commits this back to Kafka. If a consumer crashes and restarts, it resumes from the last committed offset — nothing is lost, nothing skipped.

Gotcha: When you commit matters. Commit before you finish processing and a crash loses the record (at-most-once). Commit after and a crash reprocesses it (at-least-once). There is no free lunch — you choose your failure mode, which is why idempotent consumers are so valued.

Durability via replication

A log on one disk is one disk failure away from data loss. Kafka replicates each partition to several brokers: one is the leader (handles all reads/writes) and the others are followers that copy the leader's log. If the leader dies, a follower that was fully caught up (an "in-sync replica") is promoted to leader.

Concretely, the standard production setting is replication factor 3 with min.insync.replicas=2. That means a partition tolerates the loss of one broker with zero data loss and zero downtime: a write needs 2 of the 3 replicas to acknowledge, so even if one replica is down, writes still succeed, and if the leader dies an in-sync follower already has every acknowledged record. Why 3 and not 2? With factor 2 and min.insync.replicas=2, a single broker failure halts writes entirely — you cannot satisfy "2 in sync" with only one survivor. Factor 3 is the smallest setup that survives one failure while staying available. A single well-tuned partition sustains on the order of tens of MB/s, and because partitions write in parallel, a cluster scales to millions of records/second simply by adding partitions and brokers.

  • acks=all: the producer waits until all in-sync replicas have the record before considering the write successful — strongest durability, higher latency.
  • acks=1: only the leader must confirm — faster, but a leader crash before replication loses the record.

Why decoupling producers from consumers matters

Because the log persists and consumers track their own offsets, producers and consumers never talk directly and never need to be online at the same time. A producer can fire records at full speed even if a consumer is slow, down, or hasn't been built yet. New consumers can be added later and replay history. This decoupling is Kafka's real architectural value: it turns a tangle of point-to-point integrations into a single shared, durable stream that any number of systems can tap.

Senior tip: Frame Kafka as "the central nervous system" of a company's data. One source of truth (the log), many independent readers, each at their own speed. That framing — not "a faster queue" — is what signals you actually understand it.

🎯 Interview questions

What's the difference between Kafka and a traditional message queue?

A queue deletes a message when it's consumed and typically delivers each message to one consumer. Kafka is an append-only log: reading doesn't delete anything, consumers track their own offset (bookmark), records persist by retention policy, and many independent consumer groups can read the same data and replay history.

Why does Kafka only guarantee ordering within a partition, not across a topic?

Each partition is an independent log on possibly different brokers, written in parallel for throughput. There is no global clock across partitions, so a total order would require coordination that kills scalability. To keep related events ordered, route them to the same partition via a key.

How do you make sure all events for one user stay in order?

Use the user's ID as the record key. Kafka hashes the key to pick a partition, so every event for that user lands in the same partition, where offsets are strictly ordered. Different users may land in different partitions, which is fine.

What happens to offsets when a consumer crashes and restarts?

It resumes from the last committed offset for its consumer group. Records after that are re-read. Whether a crash causes reprocessing or loss depends on whether you commit before or after processing — the classic at-least-once vs at-most-once tradeoff.

How does Kafka avoid data loss when a broker dies?

Each partition is replicated to multiple brokers with one leader and several follower replicas. With acks=all, a write is acknowledged only after all in-sync replicas have it; if the leader fails, an in-sync follower is promoted, so no acknowledged record is lost.

Why is replication factor 3 with min.insync.replicas=2 the common default?

It is the smallest configuration that survives one broker failure while staying both durable and available. A write needs 2 of 3 replicas, so one broker can be down and writes still succeed with no data loss. Replication factor 2 would block writes entirely on a single failure (you can't meet "2 in sync" with one survivor).

How does a single topic reach millions of records per second?

By partitioning. Each partition is an independent log on a (possibly different) broker writing sequentially, which disks handle extremely fast. Throughput scales roughly linearly with partition count, so you add partitions and brokers until you hit the target rate.

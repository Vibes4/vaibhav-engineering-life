How Airbnb Works Case Study

Airbnb is a two-sided marketplace connecting hosts (who list places) with guests (who book them). The interesting system-design problem is search and booking: find available places matching a location and a date range, rank them, and — crucially — guarantee that two guests can never book the same place for overlapping nights.

Figure: search path (gateway → Search service → Elasticsearch geo index) and booking path (gateway → Booking service → Bookings DB under a row lock); listing changes flow asynchronously into the search index.

Requirements (functional & non-functional)

Separate what the system does from how well it must do it.

  • Functional: hosts create listings with availability; guests search by location + check-in/check-out dates; guests book; the system blocks double-bookings.
  • Non-functional: search must be fast and geo-aware; booking must be strongly consistent (no double-booking, ever); high availability; handle bursty demand for popular listings/dates.
Senior tip: Notice search and booking have opposite consistency needs. Search can be eventually consistent (a stale result is annoying, not catastrophic). Booking must be strongly consistent (a double-booking is a real-world disaster). Design each path to its own requirement.

The core hard problem: preventing double-booking

The heart of the system is an interval-overlap check. A booking is a date range [checkIn, checkOut). A new booking for a listing is only allowed if it overlaps with none of that listing's existing confirmed bookings.

Two ranges [aStart, aEnd) and [bStart, bEnd) overlap if and only if aStart < bEnd AND bStart < aEnd. This one line is the whole game — memorize it. Note we use half-open ranges so a checkout day can be someone else's check-in day.

Gotcha: Checking "is it available?" and then "write the booking" as two separate steps creates a race condition: two guests both read "available" before either writes. You must make the check-and-write atomic (a DB transaction with a row lock, or a unique constraint), or you will sell the same night twice.

Step-by-step request flow

Walk through the critical operation — a guest booking a listing — using the components in the diagram. This is where double-booking must be prevented.

  1. Search first: the Client sends a search ("Lisbon, 12–15 Aug") over HTTPS through the API Gateway to the Search service, which runs a geo + availability query against the Elasticsearch index and returns ranked candidates in tens of milliseconds.
  2. Guest picks one: the client sends a booking request for a specific listing and date range to the Booking service via the gateway.
  3. Open a transaction: the Booking service begins an ACID transaction on the Bookings DB and issues SELECT ... FOR UPDATE on that listing's booking rows, acquiring a row lock. Any concurrent booking for the same listing now blocks here.
  4. Overlap check: inside the lock, it checks the candidate range against existing confirmed bookings using aStart < bEnd AND bStart < aEnd. If any overlap exists, it aborts and returns "no longer available".
  5. Commit: if clear, it inserts the booking row and commits, which releases the lock. The check and the write happened atomically, so the second guest — who was blocked at step 3 — now sees the new row and is correctly rejected.
  6. Propagate: the new booking eventually updates the listing's availability calendar, which is pushed asynchronously into the Elasticsearch index so future searches reflect it.
Why the lock spans the whole transaction: if you released the lock between the check (step 4) and the write (step 5), two transactions could both pass the check before either writes. Holding the row lock from FOR UPDATE through COMMIT is exactly what serializes competing bookings on the same listing.

Data model / storage choice

Listings (relational)Host, location (lat/lng), price, amenities. Source of truth, queried by search.
Bookings (relational, transactional)listing_id + date range + status. The overlap check and write happen inside one ACID transaction so consistency is guaranteed.
Search index (e.g. Elasticsearch)Denormalized copy of listings optimized for geo + filter queries. Updated asynchronously from the source of truth.

Why a separate search index? A relational DB is great for the transactional booking write but poor at "find all listings within 5km of these coordinates, available these dates, sorted by relevance". A purpose-built search engine handles geo and ranking far better.

Geo + date search

Search has two filters that work differently:

  • Geo: use a spatial index (geohash or quadtree) so "near these coordinates" is a fast prefix/range lookup instead of computing distance to every listing on Earth.
  • Dates: filter out listings already booked for the requested range. Often precomputed as an availability calendar per listing so the check is a quick lookup.
Senior tip: Do the cheap, highly-selective filter first. Geo usually eliminates 99.99% of listings instantly, leaving a small candidate set on which the more expensive availability and ranking work is cheap. Filter order is a real performance lever.

Put numbers on it. Say there are ~7M active listings worldwide. A search bounded to a city neighborhood (a geohash prefix covering a few km²) typically narrows that to a few hundred to a few thousand candidates — a 1000x–10000x reduction before any date or ranking work runs. The search service should answer in a <100ms budget: the geo lookup is a sub-millisecond index range scan, leaving headroom for the availability filter and scoring. Search traffic is bursty and read-only — easily tens of thousands of QPS at peak — which is exactly why it lives on a horizontally scalable, replicated Elasticsearch tier instead of the transactional DB.

Ranking listings

Once you have available candidates, order them. Ranking signals include price, review score, distance from the searched area, host responsiveness, and personalization. Like Reddit's "Hot", this is a blended score — but here it directly drives revenue, so it is heavily A/B tested and tuned.

Scaling and tradeoffs

Pessimistic locking (SELECT ... FOR UPDATE)Lock the listing's booking rows during the transaction. Simple and safe; can reduce throughput under heavy contention on one hot listing.
Unique constraint / exclusionLet the database reject overlapping ranges (e.g. Postgres exclusion constraint). Very safe; the loser just gets an error to retry.

Scale and the why behind the choice. Bookings are far rarer than searches — perhaps a few hundred writes/sec globally — so the booking path can afford strong consistency that a high-QPS read path could not. Contention is also local: the row lock only serializes bookings for the same listing, so two guests booking different homes never block each other, and lock hold time is just the few milliseconds of one transaction. The danger is a "hot" listing (a Super Bowl weekend) where many guests race for the same nights; there the unique/exclusion constraint shines because the loser simply gets a retryable error instead of waiting on a contended lock. This is the deliberate inverse of the search tier: partition consistency by path — relax it where reads dominate, enforce it where money and real-world allocation are at stake.

Gotcha: Don't try to enforce no-double-booking purely in application code across multiple servers — without a single point of truth (the DB transaction/constraint), concurrent app instances will still collide. Let the database be the referee.

🎯 Interview questions

What's the exact condition for two date ranges overlapping?

[aStart, aEnd) and [bStart, bEnd) overlap iff aStart < bEnd AND bStart < aEnd. Using half-open intervals lets a guest check out on the same day another checks in without a false conflict.

How do you prevent two guests booking the same listing for the same nights?

Make the overlap check and the booking insert atomic: do them in one DB transaction with a row lock (SELECT ... FOR UPDATE), or rely on a database exclusion/unique constraint that rejects overlapping ranges. Never check-then-write as two separate, racy steps.

Why separate the search index from the bookings database?

They have different jobs. The bookings DB needs ACID transactions for consistency; search needs fast geo and ranking queries. A dedicated search engine, updated asynchronously, serves reads well without burdening the transactional store.

Search and booking have different consistency requirements — explain.

Search can be eventually consistent: a momentarily stale listing is acceptable. Booking must be strongly consistent: a double-booking is a real-world failure. You design each path to its own requirement rather than forcing one model on both.

How do you make geo + date search fast at scale?

Use a spatial index (geohash/quadtree) so "near these coordinates" is a prefix/range lookup, and apply the most selective filter (usually geo) first to shrink the candidate set before running the cheaper availability check and ranking.

Walk me through the exact sequence of a booking that prevents a double-book.

Begin a transaction on the Bookings DB; SELECT ... FOR UPDATE on the listing's rows to take a row lock; run the overlap check inside the lock; if clear, insert the booking and commit (releasing the lock). A competing request for the same listing blocks at the FOR UPDATE until commit, then sees the new row and is rejected. Check and write are one atomic unit.

Why can the booking path afford strong consistency when the search path can't?

Because of volume and locality. Bookings are rare (hundreds/sec) and the lock only serializes the same listing, so the cost of strong consistency is tiny and contention is local. Search is tens of thousands of QPS and read-only, so forcing it through a transactional store would be far too expensive — you partition consistency by path.

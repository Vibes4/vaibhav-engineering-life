Lifecycle: Middleware, Guards, Interceptors, Pipes NestJS

Nest decomposes cross-cutting concerns into four distinct, ordered hooks. Knowing the exact order and what each is for is a top NestJS interview question.

The order (memorize this)

Middleware → Guards → Interceptors(pre) → Pipes → Route Handler → Interceptors(post) → Exception Filters

1. Middleware

Runs first, like Express middleware (it is Express middleware). No access to the handler/class context. Use for logging, raw request tweaks, CORS, helmet.

2. Guards (CanActivate)

Decide if the request may proceed — authentication & authorization. Return true/false (or throw). This is where JWT/roles checks live.

@Injectable()
export class AuthGuard implements CanActivate {
  canActivate(ctx: ExecutionContext): boolean {
    const req = ctx.switchToHttp().getRequest();
    return Boolean(req.headers.authorization);   // allow/deny
  }
}

3. Interceptors (NestInterceptor)

Wrap the handler — run logic before and after it (AOP-style). Use for logging timing, transforming responses, caching, mapping to a standard envelope. Built on RxJS.

intercept(ctx, next) {
  const start = Date.now();
  return next.handle().pipe(tap(() => console.log(`${Date.now() - start}ms`)));
}

4. Pipes

Transform & validate input just before the handler. The global ValidationPipe + class-validator DTOs is the standard pattern; ParseIntPipe coerces types.

Mnemonic: Guards = "can you?" (auth). Pipes = "is your input valid?" (validation/transform). Interceptors = "wrap around" (before/after). Filters = "something broke" (errors).
Binding scope: each can be applied method-, controller-, or global-scoped (app.useGlobalGuards(), @UseGuards(), etc.). Global validation + auth is the common setup.

🎯 Interview questions

What's the execution order?

Middleware → Guards → Interceptors(before) → Pipes → Handler → Interceptors(after) → (Exception Filters on error).

Guard vs Middleware?

Middleware is generic Express-level pre-processing with no handler context. Guards run later, have the ExecutionContext (know the target handler/class), and exist specifically to allow/deny — i.e. auth.

What are interceptors for?

Wrapping the handler to run before/after logic: logging/timing, response transformation, caching, serialization — AOP via RxJS.

How is validation done?

Pipes — typically the global ValidationPipe with class-validator-decorated DTOs; ParseIntPipe/ParseUUIDPipe for param coercion.

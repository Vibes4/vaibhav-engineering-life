How Spotify Works Case Study

Spotify streams audio to hundreds of millions of users while keeping playback instant and gap-free on shaky mobile networks. The interesting system-design problems are three: delivering bytes fast via a CDN, smoothly adapting quality to changing bandwidth with adaptive bitrate (ABR), and serving a massive catalog + recommendations read-heavy workload.

Figure: metadata comes from the API service over the catalog DB, audio chunks stream from CDN edge servers (the star), and recommendations are served precomputed.

Step-by-step request flow

  1. Client player opens a track; it first needs metadata and a set of chunk URLs.
  2. Metadata / API service returns track titles, artists, and playlist data, reading from the heavily cached, replicated Catalog DB.
  3. CDN Edge (the star) serves the actual audio chunks from a server physically near the user; the player runs adaptive bitrate selection per chunk against its buffer.
  4. Recommendation service supplies home-screen and "up next" picks as a fast lookup of results computed offline in batch/ML pipelines.
  5. The player keeps requesting the next chunk just-in-time from the CDN, stepping bitrate up or down between chunks to avoid an empty buffer.

Requirements: it's a read-heavy media system

  • Functional: search/browse catalog, stream a track, build playlists, recommend music, work offline (download).
  • Non-functional: low startup latency (music must begin in <1s), no rebuffering/stutter, scale to huge concurrent listeners, and handle a read:write ratio that is overwhelmingly reads (you play tracks far more than you upload them).
Senior tip: Recognize the shape of the problem: this is a read-heavy, latency-sensitive media-delivery system. That immediately points to caching everywhere (CDN for audio, caches for metadata) and pre-computation (recommendations computed offline, not per-request).

Audio delivery: why a CDN?

Audio files are large and the same popular tracks are requested by millions. Serving them all from one origin would saturate bandwidth and add latency for distant users. A CDN (Content Delivery Network) caches the audio on edge servers physically close to listeners.

  • Edge caching: the first listener in a region pulls a track to the nearby edge; everyone after gets it from there. Popular tracks have near-100% cache hit rates.
  • Segmentation: a track is split into small chunks (a few seconds each) so the player fetches just-in-time and can switch quality between chunks.
  • Lower latency & origin offload: shorter network path for users, and the origin is shielded from the read flood.

Putting numbers on it: the product target is startup latency under ~1 second and effectively zero rebuffers, so chunks are typically ~5–10 seconds of audio (e.g. ~200 KB at 320 kbps) and the player keeps a ~20–30 second buffer ahead. Because the head of the catalog is so skewed — a tiny fraction of tracks accounts for the bulk of plays — edge cache hit rates exceed ~95%, which is why a CDN is the right tool: it turns a hundreds-of-millions-of-concurrent-streams problem into mostly local reads, sparing the origin and shrinking the network round-trip to a nearby PoP (single-digit to low-tens of milliseconds). Metadata, by contrast, tolerates eventual consistency — a newly added track showing up a few seconds late is harmless — so it can be aggressively cached and replicated rather than held to the strict consistency a payment ledger would demand.

The core hard problem: adaptive bitrate (ABR)

Networks fluctuate — full bars then a tunnel. If you always streamed the highest quality, weak networks would constantly stall (rebuffer). If you always streamed the lowest, strong networks would waste their capacity on poor audio. ABR picks, per segment, the highest bitrate the current conditions can sustain.

Each track is pre-encoded at several bitrates — a bitrate ladder (e.g. 96 / 160 / 320 kbps). For each upcoming segment the player asks: which rung can I afford right now?

Throughput-based ABRPick bitrate ≤ recently measured bandwidth. Reacts fast, but bandwidth estimates are noisy, so it can oscillate or stall on sudden drops.
Buffer-based ABRDecide using how many seconds are buffered: lots of buffer → push quality up; low buffer → drop quality to avoid an empty buffer (a stall). More stable; the modern default (e.g. Netflix's BBA).
Gotcha: An empty buffer = a rebuffer = the music stops — the single worst experience. So ABR is deliberately conservative: it would rather play a slightly lower bitrate than risk silence. Maximizing average quality is secondary to never stalling.

Buffer-aware selection (what the demo shows)

A robust rule combines both signals: estimate what bandwidth allows, but gate it by buffer health. If the buffer is dangerously low, drop to the safest rung regardless of a rosy bandwidth reading; if the buffer is healthy, allow the highest rung that fits. The demo implements exactly this.

Catalog metadata and recommendations

Separate from audio bytes is the metadata (track titles, artists, playlists) and recommendations.

  • Metadata store: read-heavy, so it's heavily cached and replicated; users tolerate slightly stale catalog data, so eventual consistency is fine.
  • Recommendations: computed offline in big batch/ML pipelines (collaborative filtering: "people who liked X also liked Y") and stored pre-computed, so a home-screen request is a fast cache lookup, not a live model run.
Senior tip: Split the system by workload shape: the audio path optimizes for raw byte throughput (CDN), the metadata path for fast key lookups (caches/replicas), and recommendations for heavy offline compute (batch pipelines). One size does not fit all three; designing them separately is the senior move.

🎯 Interview questions

Why use a CDN for audio streaming?

Popular tracks are requested by millions, and audio files are large. A CDN caches segments on edge servers near users, so most requests are served from a nearby cache instead of a distant origin. This cuts latency (shorter network path), prevents the origin from being overwhelmed, and saves bandwidth — exactly what a read-heavy media system needs.

What is adaptive bitrate streaming and why is it needed?

Each track is pre-encoded at several bitrates (a ladder). The player selects, per segment, the highest bitrate the current network and buffer can sustain. It's needed because real networks fluctuate: a fixed high bitrate would stall on weak networks, and a fixed low one would waste good networks. ABR keeps playback smooth while maximizing quality.

Throughput-based vs buffer-based ABR — what's the difference?

Throughput-based picks a bitrate at or below recently measured bandwidth; it reacts quickly but is sensitive to noisy estimates and can oscillate. Buffer-based decides from how many seconds are buffered — high buffer pushes quality up, low buffer drops it to avoid a stall. Buffer-based is more stable, so modern players favor it, often blending both.

Why is never stalling more important than maximizing bitrate?

An empty buffer stops the music — a rebuffer — which users hate far more than a momentary dip in audio quality. So ABR is intentionally conservative, prioritizing buffer safety over squeezing out the highest possible bitrate. Average quality is optimized only after the no-stall constraint is satisfied.

How are recommendations served fast at Spotify's scale?

They're pre-computed offline in batch/ML pipelines (e.g. collaborative filtering) rather than generated live per request. The results are stored, so when you open the app it's a quick cache lookup instead of an expensive on-the-fly model run. This trades freshness (recommendations update periodically) for very low read latency, which suits the use case.

Why separate the metadata path from the audio path?

They have completely different workload shapes. Audio is large, immutable, and highly skewed toward popular tracks, so it wants a CDN optimized for raw byte throughput and edge caching with 95%+ hit rates. Metadata is small, frequently queried by key, and only needs eventual consistency, so it wants in-memory caches and read replicas tuned for fast lookups. Forcing both through one tier would compromise each; splitting them lets you scale and cache each according to its own access pattern.

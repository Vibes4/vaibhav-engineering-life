How LLMs Like ChatGPT Work Case Study

A Large Language Model (LLM) like ChatGPT does one deceptively simple thing over and over: predict the next token. Given the text so far, it outputs a probability distribution over what comes next, picks one, appends it, and repeats. This is autoregressive generation. The system-design interest is everything around that loop — tokenization, the transformer doing the prediction, how sampling controls creativity, the finite context window, and how to serve it cheaply with batching and a KV-cache.

Figure: the prompt is tokenized once, then each step runs the transformer (reusing its KV-cache) → sampler → appended token, looping back into the model until a stop token, after which the IDs are detokenized to text.

Tokenization: text becomes numbers

Models don't see characters or words; they see tokens — integer IDs for common chunks of text. "tokenization" might split into token + ization. A scheme like Byte-Pair Encoding (BPE) learns a vocabulary of ~50k–100k frequent sub-words so any text (even typos, code, emoji) can be represented.

  • Why sub-words, not words? A fixed word vocabulary can't handle unseen words; pure characters make sequences too long. Sub-words are the sweet spot: ~1 token ≈ 0.75 English words.
  • Why it matters: you pay per token, the context window is measured in tokens, and the model's whole world is token IDs in, token IDs out.

Step-by-step request flow

Following the pipeline in the diagram, here is one response being generated:

  1. Prompt: your raw text (system + conversation + new message) arrives as a string.
  2. Tokenizer: BPE splits it into sub-word tokens and maps them to integer IDs — done once for the whole prompt.
  3. Transformer (★): the model runs self-attention over the sequence and emits a logit (score) for every token in the vocabulary. On the first pass it reads the whole prompt (prefill) and fills the KV-cache.
  4. Sampler: temperature scales the logits, top-k/top-p trims the candidate set, and one token is sampled from the resulting distribution.
  5. Autoregressive loop (dashed): the sampled token is appended and fed back into the transformer. Thanks to the KV-cache, this decode step only computes K/V for the one new token, not the whole history. Repeat until a stop token or length limit.
  6. Detokenizer → Text: the accumulated token IDs are mapped back to characters and streamed out as readable text.

The core loop: autoregressive next-token prediction

This is the heart of the demo. Generation is a loop:

  • 1. Feed the current token sequence into the model.
  • 2. It outputs a score (logit) for every token in the vocabulary — a probability distribution over "what comes next".
  • 3. Sample one token from that distribution.
  • 4. Append it to the sequence and go back to step 1, until you emit a stop token or hit a length limit.

The demo uses a tiny n-gram (Markov) model instead of a transformer, but the loop is identical: build a next-token distribution from context, sample, append, repeat. "Just predicting the next token" is genuinely how fluent paragraphs emerge.

Senior tip: the only fundamental difference between the toy n-gram and a real LLM is how the next-token distribution is computed. The n-gram looks up counts of the last word or two; the transformer uses attention over the entire context to produce a far richer distribution. The generation loop is exactly the same.

The transformer at inference time

The model that produces the distribution is a transformer. Its key mechanism is self-attention: for each position, it weighs how much every earlier token should influence the prediction here. That's how it captures long-range meaning ("it" refers to the noun three sentences back). At inference it runs forward only — no learning — turning the input tokens into that next-token distribution.

Sampling: temperature & top-k

Step 3 — how you pick from the distribution — controls personality. Always taking the single most likely token (greedy) is repetitive and robotic; sampling adds variety.

TemperatureScales the distribution before sampling. Low (→0) = sharp, deterministic, "safe". High (>1) = flatter, more random/creative, risk of nonsense.
Top-kOnly consider the k most likely tokens, then sample among them. Prevents picking absurd low-probability tokens.
Top-p (nucleus)Consider the smallest set of tokens whose probabilities sum to p. Adapts the candidate set to how confident the model is.
Gotcha: high temperature feels "more creative" but can derail into incoherent text; temperature 0 is reproducible but bland and can loop. There's no single "best" — it's a knob you tune per use case (code completion wants low, brainstorming wants higher).

The context window

The model can only "see" a fixed number of tokens at once — the context window (e.g. 8k, 128k, 1M tokens). It includes your prompt plus everything generated so far. Run past it and the earliest tokens must be dropped or summarised — the model literally cannot attend to text outside the window.

  • Why finite? Attention cost grows with sequence length, so longer windows cost more compute and memory.
  • Consequence: "it forgot what I said earlier" usually means earlier turns fell out of the window.

Serving: batching & the KV-cache

Generation is expensive and sequential (token N+1 needs token N). Two tricks make serving efficient:

  • KV-cache: self-attention recomputes "keys" and "values" for all previous tokens every step. Instead, cache them after the first pass and reuse — so each new token only computes its own K/V, not the whole history. This turns generation from O(n²) per step into roughly O(n).
  • Batching: GPUs are most efficient processing many sequences at once. Servers group concurrent requests (continuous/in-flight batching) to keep the GPU saturated and drive down cost per token.

The numbers explain the user experience. A response is split into prefill (process all N prompt tokens in one parallel pass) and decode (generate one token at a time). Prefill is compute-bound and scales with prompt length — a 32k-token prompt does far more work before the first token appears, which is why "time to first token" climbs with context size and is often quoted separately (e.g. tens to low hundreds of ms for short prompts). Decode is memory-bandwidth-bound and runs at a steady per-token latency (commonly tens of ms, i.e. roughly 20–100 tokens/sec for a large model), which is the speed you see text streaming. Without the KV-cache, generating the 1,000th token would re-process all 999 prior tokens — O(n²) total — so the cache is what keeps decode roughly linear and makes long outputs affordable. Continuous batching then packs dozens of concurrent sequences into each GPU step; since decode is bandwidth-bound, adding more sequences raises total throughput (tokens/sec across users) with little extra per-step latency, which is why cost-per-token falls sharply as utilization rises.

Senior tip: the KV-cache is why the first token of a response is slow (the "prefill" reads the whole prompt) but subsequent tokens stream out quickly (the "decode" phase reuses the cache). That prefill-vs-decode split shapes how LLM serving systems are designed and priced.

🎯 Interview questions

What does "autoregressive" mean for an LLM?

It generates one token at a time, feeding each newly produced token back in as input for the next step. The next token is always conditioned on everything generated so far, so output is built left-to-right in a loop until a stop token or length limit.

Why tokens instead of words or characters?

Words can't handle unseen terms and need a huge vocabulary; characters make sequences too long and slow. Sub-word tokens (e.g. BPE) cover any text with a manageable vocabulary while keeping sequences short — about 1 token per 0.75 English words.

What do temperature and top-k actually control?

Both shape how the next token is sampled. Temperature scales the distribution: low makes it sharp/deterministic, high makes it flat/random. Top-k restricts sampling to the k most likely tokens, preventing rare nonsense tokens from being chosen.

What is the context window and what happens when you exceed it?

It's the fixed maximum number of tokens (prompt + generation) the model can attend to at once. Exceed it and the earliest tokens are dropped or summarized — the model can no longer "see" them, which is why long conversations seem to forget earlier details.

What problem does the KV-cache solve?

Without it, each generated token re-derives the attention keys/values for the entire history — O(n²) work. The KV-cache stores those per-token K/V vectors after computing them once, so each new step only computes its own, making decoding far cheaper and enabling fast token streaming.

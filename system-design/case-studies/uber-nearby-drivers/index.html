How Uber Finds Nearby Drivers Case Study

When you open Uber, the app must answer one deceptively simple question in a few milliseconds: "which drivers are within a couple of kilometres of me, right now?" The hard part is that drivers are moving — millions of them — each pinging their GPS location every few seconds. This is a real-time geospatial proximity search problem at massive scale, and a naive solution falls over instantly.

Figure: a write-heavy update path feeds the in-memory geo index; the rider's query reads a 3x3 block of cells.

Requirements (functional & non-functional)

Pin down what the system must do (functional) versus how well it must do it (non-functional) before touching any data structure.

  • Functional: drivers continuously publish their location; a rider can query "drivers within X km of (lat, lng)"; the system returns a short list of candidate drivers to dispatch.
  • Non-functional: very low query latency (<100ms), an enormous write rate (every driver updates location every 4-5s), eventual consistency is fine (a driver one second stale is acceptable), high availability over perfect accuracy.
Senior tip: Notice the write rate dwarfs anything else — millions of location updates per second. That single fact tells you the location index must be cheap to update and must not require re-sorting the entire world on every ping.

Step-by-step request flow

  1. Drivers emit a GPS ping every 4-5 seconds. At ~5M active drivers that is roughly 1M location writes per second — the dominant load on the whole system.
  2. The Location-update service receives each ping, computes the driver's grid cell (geohash/H3) and forwards the upsert.
  3. The Geo Index (the star) does an O(1) update: move the driver from their old cell bucket to the new one. No global re-sort, which is the only way to absorb the write firehose.
  4. A Rider app issues a "drivers near (lat,lng)" query to the Matching service.
  5. The Matching service computes the rider's cell and reads that cell plus its 8 neighbours from the Geo Index, getting a few dozen candidates.
  6. It returns a ranked candidate list after the fine step (exact distance/ETA), and dispatch proceeds.

The core hard problem: proximity search on moving points

The intuitive approach — store every driver's lat/lng and, for each rider query, compute the distance to all drivers and keep the close ones — is O(N) per query. With millions of drivers that is millions of distance calculations per request. It is hopeless.

The key insight: you don't need exact distance to find candidates. You first need a cheap way to throw away the 99.99% of drivers who are obviously far away (different city, different neighbourhood), and only then do precise math on the handful that remain. That "throw away the far ones cheaply" step is what a spatial index does.

Gotcha: A plain database index on lat and lng as two separate columns does not solve this. A range query like "lat between A and B AND lng between C and D" can only use one column index efficiently; the other becomes a scan. Geospatial proximity needs an index that combines both dimensions into one sortable key.

Spatial indexing options

The whole game is mapping 2D coordinates onto a 1D key (or a grid) so that "near in space" becomes "near in the index". Three popular approaches:

Geohash (grid cells)Interleave the bits of lat and lng into one string. Each character refines the cell. Nearby points usually share a prefix, so a prefix scan finds neighbours. Simple, but rectangular cells distort near the poles and the 8 neighbours must be computed explicitly.
QuadTreeRecursively split a region into 4 quadrants, splitting further only where drivers are dense. Adapts to population density (dense downtown vs empty desert), but the tree must be rebalanced as drivers move, which is extra bookkeeping.
H3 (hexagons, used by Uber)Tile the globe in hexagons. Every neighbour is exactly the same distance away (hexagons have uniform adjacency, unlike squares whose diagonal neighbours are farther). Cleanest "ring of neighbours" math, which is why Uber built and open-sourced it.

Geohash-style bucketing (the demo)

The simplest version, which our demo implements: quantize each driver's position into a fixed grid cell, keyed by something like "cellX:cellY". To handle a location update, you just recompute one cell key and move the driver between buckets — cheap, O(1), no global re-sort. That directly answers the brutal write-rate requirement.

To answer "nearby", you compute the rider's cell, then scan that cell plus its 8 surrounding cells (a 3x3 block). Why the neighbours? Because a driver just across a cell boundary is physically close to the rider but sits in an adjacent bucket. Checking only the rider's own cell would miss them.

Gotcha: Cell size is a real tradeoff. Make cells too big and each query returns thousands of candidates (you're back to scanning a lot). Make them too small and a "nearby" radius spans many cells, so you must scan far more than 9 of them. Tune cell size to roughly the query radius.

Storage & scaling the write path

Driver locations are hot, ephemeral, and written constantly — a perfect fit for an in-memory store like Redis (which has native geo commands) rather than a disk-backed relational database. Put concrete numbers on it: ~5M concurrent drivers each pinging every 4-5s is on the order of 1M writes/second. A disk-backed B-tree simply cannot sustain that, and it does not need to — the data is disposable. If the whole index were lost, every driver re-reports its position within 5 seconds and the index self-heals. That is the why behind keeping it in memory: durability is irrelevant, so you trade it away for raw write speed and pay nothing.

  • Shard by region: partition the index by city or geographic region. A rider in Mumbai never needs the Tokyo index, so those shards live on different servers and scale independently.
  • Keep the source of truth small: the location index only needs current position. Historical trails go to a separate analytics pipeline, not the live index.
  • Absorb updates with a queue: location pings can flow through a stream (e.g. Kafka) so a burst of updates doesn't overwhelm the index directly.

Tradeoffs & consistency

Uber happily accepts eventual consistency on driver positions. If a driver's shown location is one second old, the rider is unaffected — the candidate list is still correct enough to dispatch. Trading perfect freshness for a cheap, fast index is the right call. The candidate set from the spatial index is also deliberately approximate: it returns "drivers in nearby cells", and a second precise step (real distance + ETA + driver availability) ranks them.

Senior tip: Always describe this as two phases — a cheap "coarse filter" (spatial index narrows millions to dozens) followed by an expensive "fine rank" (exact distance/ETA on the survivors). Conflating them is the mistake; separating them is what makes the system fast.

🎯 Interview questions

Why can't we just compute the distance from the rider to every driver?

That is O(N) per query over millions of moving drivers — millions of distance calculations per request. It cannot meet a sub-100ms latency target. You need a spatial index to cheaply discard the far-away majority first, then do precise math only on the survivors.

Why scan the 8 neighbouring cells and not just the rider's own cell?

A driver standing just across a cell boundary is physically close to the rider but lives in an adjacent bucket. Querying only the rider's cell would miss those nearby drivers. Scanning the 3x3 block around the rider covers all drivers who could be within roughly one cell's distance.

How do you choose the grid cell size?

Match it to your typical query radius. Too large and each query returns too many candidates (you're back to scanning lots). Too small and a single radius spans many cells, so you scan far more than 9. The cell should be about the size of the search radius.

Why does a plain two-column (lat, lng) database index not work?

A B-tree index can only efficiently use one column for a range scan; the second dimension degrades to a scan. Proximity needs both dimensions folded into a single sortable key — that's what geohash bit-interleaving (or a quadtree/H3) achieves.

How does the system handle millions of location updates per second?

Updating the index is O(1): recompute one cell key and move the driver between buckets — no global re-sort. Store it in memory (Redis), shard by region so cities scale independently, and optionally buffer the firehose of pings through a stream before applying them.

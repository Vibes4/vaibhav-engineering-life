How the Stock Exchange Works Case Study

A stock exchange is, at its heart, a giant matching engine: millions of people send buy and sell intentions, and the system must pair them up fairly and instantly. The interesting system-design problem is doing this with microsecond latency, perfect fairness (who got there first?), and zero tolerance for losing or duplicating a single trade — all at once.

Figure: orders flow through a gateway and sequencer into the single in-memory matching engine, which journals every event and emits trade and market-data feeds.

Step-by-step request flow

  1. Order Gateway receives an order, authenticates the trader, validates the symbol/price/size, rate-limits, then forwards it inward. Many gateways run in parallel.
  2. Sequencer merges all gateways' orders into one totally ordered stream and stamps each with a monotonic sequence number — this single point is what makes fairness (price-time priority) well-defined.
  3. Matching Engine (the star) reads that stream on a single thread and applies each event to the in-memory order book (bids/asks), matching crossing orders and generating fills.
  4. Journal records every event append-only and replicated before it is applied, so the exact book can be rebuilt by replay after a crash.
  5. Trade Feed publishes executions to participants and downstream clearing/settlement.
  6. Market-Data Feed broadcasts the updated book state and last price to millions of subscribers via a separate read-optimized pipeline.

Requirements: what must it actually do?

Before designing anything, separate what the system does (functional) from how well it must do it (non-functional). Confusing these is the #1 fresher mistake.

  • Functional: accept orders (buy/sell, limit/market), maintain an order book per symbol, match crossing orders, broadcast trades and price updates, allow cancellation.
  • Non-functional: ultra-low latency (often <100 microseconds end-to-end), determinism (same inputs → same trades, always), fairness (strict ordering), durability (a confirmed trade is never lost), and high throughput (millions of orders/second at peak).

Putting numbers on it: a top-tier matching engine processes a single order in roughly 1–10 microseconds in-core, and the wire-to-wire ("tick-to-trade") path is budgeted at tens of microseconds. NASDAQ's INET and the LMAX Disruptor are both engineered around this: LMAX famously demonstrated ~6 million orders/second on one thread by eliminating locks and cache misses. Why so extreme? Because latency is a competitive product — a market maker quoting a price needs to cancel and re-quote faster than a rival can pick off a stale quote. A few microseconds of jitter is the difference between a profitable and a losing strategy, so exchanges co-locate clients in the same data center and measure latency in nanoseconds of variance.

Senior tip: Notice that "availability" is NOT the top priority here, unlike a typical web app. Correctness and fairness beat uptime — an exchange would rather halt trading than execute trades in the wrong order. Knowing which property to sacrifice is the whole game.

The core hard problem: the matching engine

The matching engine is the beating heart. It holds the order book — two sorted lists per symbol: bids (buyers, sorted highest price first) and asks (sellers, sorted lowest price first). A trade happens when the best bid price meets or exceeds the best ask price (they "cross").

The tie-breaking rule is price-time priority:

  • Price first: a buyer willing to pay more gets served before a buyer offering less. Best price wins.
  • Time second: among orders at the same price, the one that arrived earliest gets filled first. First come, first served.

Think of it like a single-file queue at a bakery, except people who offer more money get to cut to the front. Within the same offer, it's pure FIFO.

Why a single thread? Determinism over parallelism

Surprisingly, the matching core for one symbol usually runs on a single thread (often pinned to one CPU core). A fresher's instinct is "parallelize for speed!" — but here that breaks fairness, because two threads could match the same order or reorder events.

Single-threaded matcherDeterministic, trivially correct ordering, no locks. Limited to ~one core's throughput per symbol, but symbols can be sharded across cores. This is what real exchanges (LMAX, NASDAQ) do.
Multi-threaded matcherMore raw throughput per symbol, but needs locks/coordination that add latency and risk non-deterministic ordering — fatal for fairness. Almost never used for the core match loop.
Gotcha: Sharding by symbol works only because orders for different symbols never interact. An order for AAPL can never match against MSFT, so each symbol is an independent matching engine. This is why exchanges scale horizontally by symbol, not by splitting one symbol's book.

Data model and storage: in-memory first

The live order book lives entirely in RAM — disk is far too slow for microsecond matching. But RAM is volatile, so durability comes from an append-only event log: every incoming order/cancel is written sequentially to a journal (and replicated) before it is processed.

  • Event sourcing: the book is never stored directly; it is derived by replaying the ordered log of events. Crash? Replay the log and rebuild the exact same book.
  • Sequential writes: appending to a log is the fastest durable write a disk can do (no random seeks).
  • Replication: the log is streamed to backup machines so a failover replays the identical sequence and gets an identical book.

Consistency and atomicity

A match must be atomic: when a buy and sell cross, both sides fill (or partially fill) together — you can never debit the seller's shares without crediting the buyer. Because matching is single-threaded and in-memory, atomicity is easy: one thread processes one order to completion before touching the next. No partial states are ever visible.

Senior tip: This is the elegance of the design — by serializing all events through one ordered log and one thread, you get atomicity, consistency, and determinism almost for free. The hard distributed-systems problems are pushed to the edges (gateways, market-data fan-out, settlement), keeping the sacred core simple.

Scaling and the rest of the pipeline

The matching core is fast but small. The surrounding system absorbs scale:

  • Gateways: many front-end servers validate, authenticate, and rate-limit orders, then forward them into the single ordered stream.
  • Market data fan-out: after each trade, the new book state and last price are broadcast to potentially millions of subscribers via a separate read-optimized pipeline (multicast / pub-sub).
  • Clearing & settlement: a trade on the exchange is just a promise; actual money/share transfer happens afterward in a separate (slower, eventually-consistent) settlement system.

🎯 Interview questions

What is price-time priority and why does it matter?

It's the matching rule: orders are filled by best price first, and among equal prices by earliest arrival time. It matters because it defines fairness — without a deterministic rule, two traders offering the same price would have no defined order, which could be exploited or disputed. It makes the engine's output predictable and auditable.

Why run the matching engine on a single thread?

To guarantee deterministic ordering and atomicity without locks. Multiple threads could reorder events or partially apply a match, breaking fairness. A single thread processes one event fully before the next, so the book is always in a consistent state. Throughput is scaled by sharding different symbols onto different cores, not by parallelizing one symbol.

How do you keep an in-memory order book durable across crashes?

Event sourcing: write every order/cancel to an append-only, replicated log before applying it. The book is a derived view, rebuilt by replaying the log. After a crash, replay the log (and the replica continues from the same sequence) to reconstruct the identical book. Sequential log writes are fast enough to stay on the latency budget.

How does the exchange scale to millions of orders per second?

Shard by symbol — each symbol is an independent matching engine, so they run on separate cores/machines with no coordination. Front-end gateways scale horizontally to handle validation and connections, and market-data distribution is a separate read-side pipeline (multicast/pub-sub) decoupled from matching.

What happens to an order that doesn't immediately match?

A limit order that doesn't cross the opposite side rests in the book at its price, joining the queue at that price level by arrival time. It stays until it's matched by a future incoming order, cancelled, or expires. A market order, by contrast, matches against the best available prices immediately and any unfilled remainder is typically cancelled rather than rested.

Why is there a separate sequencer in front of the matching engine?

Multiple gateways accept orders concurrently, so their arrival order is ambiguous. The sequencer merges every order into one totally ordered stream with monotonic sequence numbers, establishing a single authoritative "who came first." That total order is what makes price-time priority deterministic and auditable, and it's also the exact order written to the journal and replayed on failover, so replicas rebuild an identical book.

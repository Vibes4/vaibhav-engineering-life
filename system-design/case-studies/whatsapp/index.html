How WhatsApp Works Case Study

WhatsApp delivers tens of billions of messages a day, and yet its own servers can't read a single one. That's the fascinating tension: it must be a reliable post office (store messages, forward them when you come back online, show ticks) while being a post office that cannot open the envelopes. This case study is about end-to-end encryption, offline store-and-forward, and the single/double/blue tick state machine.

Figure: the server is a blind relay that store-and-forwards ciphertext; the keys that can decrypt it live only on the two endpoint devices.

Requirements (functional & non-functional)

  • Functional: send a message to another user, deliver it even if they're currently offline, show the sender whether it was sent, delivered, and read.
  • Non-functional: messages must be readable only by sender and recipient (end-to-end encrypted), delivery must be reliable and ordered, and the system must scale to billions of users with low latency.

The defining constraint is the security one: the server is untrusted. We design as if the server (and anyone who hacks it) is an eavesdropper. That single rule shapes the entire architecture.

Step-by-step request flow

Tracing the three boxes in the diagram, one message travels like this:

  1. Sender device (a star): Alice's app encrypts the plaintext with the symmetric key derived from the shared secret. The plaintext and the private key never leave her phone.
  2. WhatsApp server: receives only ciphertext. If Bob is online it forwards instantly; if he's offline it persists the ciphertext in his queue and (optionally) fires a push notification. It is structurally blind to content.
  3. Recipient device (a star): Bob's app pulls the ciphertext on reconnect and decrypts it locally with his own key material. Only here does plaintext exist again.
  4. ACK return path: Bob's device sends delivery- and read-ACKs back through the server to Alice. These are routing metadata, not encrypted content, and drive the tick state machine.

The core hard problem: end-to-end encryption without sharing a password

If Alice and Bob want a shared secret key to encrypt messages, the obvious idea is "Alice picks a key and sends it to Bob." But the server relays that message — so the server would learn the key and could read everything. We need Alice and Bob to agree on the same secret while only ever sending things over the wire that an eavesdropper can't turn into the secret.

This is solved by Diffie-Hellman key exchange. Each side generates a private key (kept secret, never sent) and a public key (derived from it, safe to send). Through the math of modular exponentiation, Alice can combine her private + Bob's public and Bob can combine his private + Alice's public, and both arrive at the identical shared secret — which never traveled across the network.

Senior tip: The intuition for Diffie-Hellman is mixing paint. Everyone agrees on a public base color. Each person mixes in a secret color and shares the mixture (you can't un-mix it). When each adds their own secret color to the other's mixture, both end up with the same final color — but an eavesdropper who only saw the two public mixtures can't reproduce it.

Encrypting the actual message

Diffie-Hellman gives us a shared secret, but it's slow and only run occasionally. The shared secret is used to derive a fast symmetric key (AES). Every message is then encrypted with AES, which is fast enough for billions of messages.

  • Asymmetric (DH): used to agree on a key safely over an open channel. Slow.
  • Symmetric (AES): used to encrypt the message bodies. Fast. Both sides already share the key, so no key needs to be sent.

The real Signal Protocol goes further with a "ratchet" that derives a new key for every single message, so cracking one message doesn't reveal past or future ones (forward secrecy). The core idea, though, is exactly the DH + symmetric combo above.

Gotcha: End-to-end encryption protects message content, not metadata. The server still must know who is talking to whom and when, to route messages. Don't claim E2EE hides everything in an interview — it hides the envelope's contents, not the addresses on it.

Offline delivery: store-and-forward

Bob is often offline (phone in pocket, no signal). The server can't just drop Alice's message. It acts as a store-and-forward mailbox: it holds the (already encrypted) message in a queue keyed to Bob, and pushes it the instant Bob reconnects.

Bob onlineServer immediately forwards the ciphertext over Bob's live connection; near-instant delivery.
Bob offlineServer persists the ciphertext in Bob's queue, optionally pings a push notification, and delivers on reconnect. Deletes it after delivery (WhatsApp doesn't keep messages long-term).

Crucially, what's stored is the ciphertext. Even sitting in the server's queue for days, the message is unreadable to the server.

The tick state machine (sent → delivered → read)

Those grey and blue ticks are a small but classic state machine driven by acknowledgements bouncing back through the server:

  • Single grey tick (sent): the server received the message from Alice. That's all Alice knows so far.
  • Double grey tick (delivered): the message reached Bob's device. Bob's app sends a delivery-ACK back through the server to Alice.
  • Two blue ticks (read): Bob actually opened the chat. His app sends a read-ACK back to Alice.

Each transition is one-way and triggered by an ACK travelling the reverse path. Note the receipts are about routing/UI and are not encrypted content — they're metadata the server legitimately handles.

Senior tip: Model receipts as an explicit state machine with idempotent transitions. ACKs can arrive late, out of order, or twice (mobile networks are flaky). "delivered" must never overwrite "read", and a duplicate ACK must be a no-op.

Tradeoffs to discuss

  • Persistent connection vs. polling: WhatsApp keeps a long-lived connection (so it can push instantly), which costs server memory per user but gives real-time delivery. The famous Erlang result: WhatsApp engineers held ~2 million concurrent connections on a single server, which is the whole reason a small team could serve hundreds of millions of users. At ~2B users, polling every few seconds would mean billions of wasted requests per minute and drained batteries; one idle socket costing a few KB of kernel memory is far cheaper than that churn, and it lets the server push a message the instant it arrives instead of waiting for the next poll.
  • How long to store undelivered messages: longer = more reliable for users who are offline for weeks, but more storage and a bigger privacy surface. WhatsApp deletes after delivery / a fixed window.
  • Read receipts privacy: nice for senders, uncomfortable for recipients — hence the option to disable blue ticks, a product tradeoff layered on the technical one.

🎯 Interview questions

How can Alice and Bob agree on a secret key when the server relays everything between them?

Diffie-Hellman key exchange. Each generates a private key (never sent) and a public key (safe to send). Alice combines her private with Bob's public; Bob combines his private with Alice's public; the math yields the same shared secret. An eavesdropper sees only the two public keys, from which deriving the secret is computationally infeasible. The secret itself never crosses the wire.

If Diffie-Hellman is so secure, why also use AES?

Asymmetric crypto like DH is computationally expensive and awkward for bulk data. So we use DH once to agree on a shared secret, derive a symmetric AES key from it, and encrypt the actual messages with fast AES. It's the standard hybrid: asymmetric to establish the key, symmetric for the payload.

What does end-to-end encryption NOT protect?

Metadata: who messaged whom, when, message sizes, and online status. The server must see this to route messages and show ticks. E2EE protects message content only. Claiming it hides everything is a common interview mistake.

How are messages delivered to an offline user?

Store-and-forward. The server queues the still-encrypted message keyed to the recipient, optionally fires a push notification, and forwards it the moment the recipient reconnects, then deletes it. The stored payload is ciphertext, so even queued messages are unreadable to the server.

Why model delivery receipts as an explicit state machine?

Because ACKs travel flaky mobile networks and can arrive late, duplicated, or out of order. An explicit state machine (sent → delivered → read) with idempotent, one-directional transitions guarantees correctness: a duplicate "delivered" is a no-op and can never regress a message already marked "read".

Why hold a persistent connection per user instead of polling, at billions of users?

A push model gives instant delivery and avoids the enormous waste of billions of empty poll requests per minute plus the battery drain on every phone. Persistent connections cost memory per user, but it's cheap per socket — WhatsApp famously sustained ~2 million concurrent connections on one Erlang server. That density is precisely what let a tiny team serve hundreds of millions of users.
